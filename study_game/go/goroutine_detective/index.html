🔍 案件 1 / 5

忘记 cancel

func handleRequest() { ctx, _ := context.WithTimeout( context.Background(), 5*time.Second, ) // ↑ cancel 函数被丢弃！ go worker(ctx) // ... 其他处理 } func worker(ctx context.Context) { for { select { case <-ctx.Done(): return default: doWork() } } }

这段代码的泄漏点是？

Acancel 函数被丢弃，context 相关资源（timer、goroutine）在超时前不会被释放
Bworker goroutine 没有退出机制，会永远运行
CWithTimeout 已设置超时，超时后 ctx.Done() 自动触发，没有泄漏
DdoWork() 函数可能阻塞，导致 goroutine 泄漏

🔧 修复方案：
ctx, cancel := context.WithTimeout(...)
defer cancel() // 必须调用！
🔍 案件 2 / 5

channel 无人关闭

func startWorkers() { jobs := make(chan int) // 启动 3 个 worker for i := 0; i < 3; i++ { go func() { for job := range jobs { process(job) } }() } // 发送任务 for _, j := range taskList { jobs <- j } // ← 发送完毕，但没有 close(jobs) }

谁应该关闭 channel？何时关闭？

A接收方（worker goroutine）负责关闭，任务处理完后关闭
B发送方（startWorkers）负责关闭，全部任务发送完毕后 close(jobs)
C不需要关闭，goroutine 会在函数返回时自动结束
D用 sync.WaitGroup 等所有 worker 结束后关闭

🔧 修复方案：
所有任务发送完毕后：close(jobs)
worker 的 for job := range jobs 会自动退出。
🔍 案件 3 / 5

WithTimeout 未感知

func fetchData(ctx context.Context) { ctx, cancel := context.WithTimeout(ctx, 3*time.Second) defer cancel() go func() { // 内部没有监听 ctx.Done()！ result := slowDBQuery() // 可能需要 10s resultCh <- result }() select { case r := <-resultCh: handleResult(r) case <-ctx.Done(): log.Println("timeout!") } }

超时后，goroutine 泄漏了吗？生命周期多长？

A没有泄漏。fetchData 返回后，内部 goroutine 会被 GC 自动回收
B泄漏。goroutine 内没有监听 ctx.Done()，会一直运行到 slowDBQuery() 返回（可能 10s）
C泄漏。resultCh 永久阻塞，goroutine 永远不退出
D没有泄漏。ctx 超时后，goroutine 自动感知并退出

🔧 修复方案：
slowDBQuery 应接受 ctx 参数，内部用 select { case <-ctx.Done(): return } 监听取消。 或者给 resultCh 设置缓冲（make(chan T, 1)），确保超时后 goroutine 能写入退出。

综合案例：HTTP Handler

func handler(w http.ResponseWriter, r *http.Request) { ctx, _ := context.WithTimeout(r.Context(), 2*time.Second) // Bug 1 ch := make(chan string) // 无缓冲 go func() { result := queryDB(r.Context()) // Bug 2: 用了原始 ctx ch <- result // Bug 3: 无缓冲，若 handler 超时退出则永久阻塞 }() select { case result := <-ch: fmt.Fprintln(w, result) case <-ctx.Done(): http.Error(w, "timeout", 504) } }

代码中有 3 个 Bug，全选出来：

🔧 三个修复：
A: ctx, cancel := ...; defer cancel()
B: queryDB(ctx)（传入带超时的 ctx）
C: ch := make(chan string, 1)（缓冲为 1，防止 goroutine 永久阻塞）
破案了！

你已揪出 5 种典型 goroutine 泄漏场景：
忘记 cancel · channel 无人关闭 · 不监听 ctx.Done() ·
context 传播方向 · 综合 HTTP Handler。
生产代码中，这些都是排行榜前列的内存泄漏原因。
