Mutex 死锁

var mu sync.Mutex func transfer() { mu.Lock() // 处理业务逻辑... mu.Lock() // ← 注意这里 // 更多操作... mu.Unlock() mu.Unlock() }

调用 transfer() 后会发生什么？

你的裁决：

理由：

⚖️ 裁判词：Go 的 sync.Mutex 不可重入（non-reentrant）。同一 goroutine 已持有锁时再次 Lock，会永久阻塞，形成单 goroutine 自我死锁。Go 运行时检测到无法推进后，抛出 "all goroutines are asleep - deadlock!"。
第 2 关

RWMutex 升级陷阱

var rw sync.RWMutex func readThenWrite() { rw.RLock() // 读操作... rw.Lock() // 在 RLock 内部升级为写锁 // 写操作... rw.Unlock() rw.RUnlock() }

这段代码会怎样？

你的裁决：

修复方案（选正确的）：

⚖️ 裁判词：RLock 内升级 Lock 必然死锁。Lock() 需要等所有读者退出，但当前 goroutine 自己就持着读锁不退出，形成自我等待环。正确做法：先 RUnlock，再 Lock（但要注意此窗口期数据可能被其他 writer 修改）。
第 3 关

WaitGroup 误用

var wg sync.WaitGroup func main() { for i := 0; i < 3; i++ { go func() { wg.Add(1) // ← Add 在 goroutine 内部 defer wg.Done() // 业务逻辑... }() } wg.Wait() fmt.Println("all done") }

这段代码可能出现什么问题？

你的裁决：

根本原因：

⚖️ 裁判词：wg.Add(1) 必须在 goroutine 启动之前调用。goroutine 启动是异步的，scheduler 可能让 Wait() 先执行——此时计数为 0，Wait 直接返回，goroutine 后续调用 Done() 则引发 panic: sync: negative WaitGroup counter。
第 4 关

Channel 关闭陷阱

4 段代码，判断各自的行为：

⚖️ 裁判词（Channel 使用铁律）：
1. 向已关闭的 channel 写入 → panic
2. 读已关闭的 channel → 立即返回零值 + ok=false，不阻塞
3. 读写 nil channel → 永久阻塞
4. 关闭已关闭的 channel → panic
记忆口诀：写/关 nil channel 阻塞，关/写 closed channel panic，读 closed 返回零值。

defer 顺序

func run() { defer fmt.Println("defer-1") defer fmt.Println("defer-2") defer func() { if r := recover(); r != nil { fmt.Println("recovered:", r) } }() panic("oops") } func main() { run() fmt.Println("after run") }

步骤一：这段代码会 panic 崩溃吗？

步骤二：选择正确的输出顺序：

⚖️ 裁判词：defer 以 LIFO 顺序执行（后注册先执行）。panic 发生后，会沿调用栈展开并依次执行各 defer。recover() 必须在 defer 直接调用的函数内才有效，它拦截 panic 后，后续 defer 继续正常执行。"after run" 会打印，因为 run() 的 panic 被 recover 了。
第 6 关

select 盲点

var ch1 chan int // nil channel ch2 := make(chan int, 1) ch2 <- 42 select { case v := <-ch1: fmt.Println("ch1:", v) case v := <-ch2: fmt.Println("ch2:", v) }

这段代码会输出什么？

你的裁决：

追加问题：如果两个 case 都 ready，select 如何选择？

⚖️ 裁判词：select 中的 nil channel 永远不会就绪，等价于该 case 不存在。因此只有 ch2 就绪，输出 42。若多个 case 同时就绪，Go 运行时随机选一个，防止饥饿（这是语言规范保证的，不是实现细节）。这种随机性是 Go 的刻意设计。
