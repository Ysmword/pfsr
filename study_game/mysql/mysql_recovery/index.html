WAL 写入顺序

一条 UPDATE 事务提交时，InnoDB 按以下顺序操作。把步骤 1-5 拖入正确位置，或直接选择正确的写入顺序：

核心机制：
• redo log：InnoDB 引擎层，记录物理变更，用于崩溃恢复
• binlog：Server 层，记录逻辑变更，用于主从复制和时间点恢复
• 两阶段提交（2PC）：保证 redo log 和 binlog 一致性

以下哪个顺序正确描述了 InnoDB 事务提交时 WAL 的写入过程？

A. ① binlog → ② redo log prepare → ③ 数据刷盘 → ④ redo log commit
B. ① redo log prepare → ② binlog → ③ redo log commit → ④ 数据页刷盘（后台）
C. ① 数据页刷盘 → ② redo log → ③ binlog → ④ 提交
D. ① redo log → ② redo log commit → ③ binlog → ④ 数据刷盘

第 4 关

持久化参数配置

innodb_flush_log_at_trx_commit 控制 redo log 刷盘时机：

值 行为 风险
0 每秒刷盘一次，提交时只写 OS 缓冲 最多丢 1 秒数据
1 每次提交都 fsync，最安全 无数据丢失风险
2 每次提交写 OS 缓冲，每秒 fsync OS 崩溃丢 1 秒数据

金融支付系统，对数据安全性要求极高，不能丢任何一条提交的交易记录。应该配置？

innodb_flush_log_at_trx_commit = 1（每次提交都 fsync）
innodb_flush_log_at_trx_commit = 2（性能更好，OS 崩溃才丢失）
innodb_flush_log_at_trx_commit = 0（性能最高）

同时，binlog 也需要配置 sync_binlog，金融场景应该设为？

sync_binlog = 1（每次提交都 fsync binlog）
sync_binlog = 0（依赖 OS 刷盘，性能最好）
sync_binlog = 100（每 100 次提交刷一次）
