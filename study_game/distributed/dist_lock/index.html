第 2 关

Redis 主节点宕机

客户端 A 用 SETNX 获取了锁，存在主节点，主节点在锁同步到从节点之前宕机，Sentinel 把从节点提升为主。

这种情况会发生什么？

A. 没问题，新 Master 会用 AOF 恢复锁数据
B. 客户端 B 可以成功获取锁，导致两个客户端同时持有锁（锁安全性破坏）
C. 客户端 B 会等到锁超时才能获取，没有问题

第 3 关

Redlock 算法

Redlock：5 个独立 Redis 节点，客户端依次向所有节点申请锁，超过半数（≥3）成功才视为获锁。

场景：5 节点 Redlock，客户端 A 已在节点 1、2、3 上获锁（3/5 超半数）。
此时节点 3 突然宕机，客户端 B 向节点 3（宕机）、4、5 申请锁，节点 4、5 没有锁记录。

客户端 B 能获取到锁吗？

A. 不能，B 只能在节点 4、5 上获锁（2/5 = 不超半数），获锁失败
B. 能，B 向 3（宕机）、4、5 申请，4 和 5 没有锁，B 可获 2 票但失败；但如果 B 向其他节点申请也算的话 B 可能获锁
C. 在极端时序下（如节点3宕机前A的锁未持久化，节点3重启后B再申请），存在 B 也能获锁的风险

第 4 关

ZooKeeper 临时节点

ZooKeeper 分布式锁基于临时顺序节点：客户端创建临时节点，最小序号的节点持锁，其他客户端 watch 前一个节点。

问题 1：持锁的客户端 A 突然宕机（进程崩溃），锁会自动释放吗？

问题 2：如果持锁客户端 A 由于 GC STW 暂停了 30 秒，导致 ZK Session 超时，锁自动释放了，但 A 以为自己还持锁并继续操作临界资源。这是什么问题？如何防止？

Q1：A 宕机，锁会自动释放吗？

会。临时节点在客户端 Session 断开时由 ZooKeeper 自动删除
不会。需要客户端主动调用 delete 才能释放
不会立即释放，等 TTL 超时才释放

Q2：GC 暂停导致锁被抢走但 A 继续操作，如何防止？

使用 Fencing Token（锁版本号）：操作资源时带上 token，资源侧拒绝旧 token 的请求
延长 Session TTL，避免 GC 期间超时
这个问题无解，ZK 锁本质上是不安全的
