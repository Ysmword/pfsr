第 1 关

特性对比

🍃
MongoDB
  • 文档模型（BSON/JSON）
  • Schema 动态灵活
  • 天然水平扩展（Sharding）
  • 嵌套文档避免 JOIN
  • 4.0+ 支持多文档事务
  • 适合非结构化/半结构化
🗄️
MySQL
  • 关系模型（表/行/列）
  • Schema 严格固定
  • 垂直扩展为主（分库分表复杂）
  • 强大的 JOIN 能力
  • 完整 ACID 事务
  • 适合结构化、强一致性场景

以下哪个说法错误？

A. MongoDB 4.0 以前不支持多文档 ACID 事务，只支持单文档原子性
B. MongoDB 不支持任何形式的索引，所有查询都是全集合扫描
C. MySQL 做水平扩展（Sharding）比 MongoDB 复杂得多
D. MongoDB 的 Schema-less 特性意味着同一集合中不同文档可以有不同字段

第 3 关

Schema 设计差异

用户的商品信息，不同商品属性差异很大（手机有 CPU/内存，衣服有尺码/颜色）。两种设计对比：

MongoDB 文档
// 手机 { name: "iPhone", cpu: "A17", storage: "256GB" } // 衣服（同集合） { name: "T-Shirt", size: ["S","M","L"], color: "red" }
MySQL 关系表
-- 方案一：宽表（空列多） products(id, name, cpu, storage, size, color, ...) -- 方案二：EAV 模式 product_attrs( product_id, attr_name, attr_value)

对于属性差异极大的多态商品场景，哪种方案最合适？

A. MongoDB 文档模型：每个商品文档只存自己的字段，灵活高效
B. MySQL 宽表：一张表存所有属性，简单但空列浪费严重，扩展难
C. MySQL EAV（属性值）模式：灵活但查询复杂、性能差、无法利用索引
D. MySQL + JSON 列：折中方案，可以，但没有 MongoDB 的聚合和索引能力强

第 4 关

综合选型

一个内容平台（类似掘金/知乎）需要存储：文章内容（结构差异大）、用户账户（严格关系）、评论（嵌套）。应该如何搭配？

A. 全用 MySQL：关系型数据库更可靠，统一管理
B. 全用 MongoDB：所有数据都是文档，统一存储
C. 混合使用：用户账户用 MySQL（ACID 事务，关系一致性），文章内容+评论用 MongoDB（Schema 灵活，嵌套文档）
D. 文章内容用 MongoDB，其他全用 MySQL，并用 MongoDB 事务同步
