SLOWLOG 检测到高频命令：
GET user:rank:top100 — 120,000 calls/min
延迟: 2ms（正常 0.1ms）
CPU 使用率飙升至 95%
单个 key 的访问量占总流量的 73%

诊断结论：这是哪种故障？

🔥 热 key（Hot Key）— 单个 key 被频繁访问，集中在某个分片
📦 大 key（Big Key）— 某个 key 的 value 体积过大，操作耗时高
💨 缓存穿透 — 大量请求查询不存在的 key，直接打穿到 DB
❄️ 缓存雪崩 — 大量 key 同时过期，请求集中打到 DB

💊 治疗方案 2 / 5

选择最优方案（可以只选一个）：

key 分片（Sharding）
将热 key 复制为 user:rank:top100:0~9，随机读取分散压力
直接删除该 key
删除热 key，让请求全部回源 DB
扩容 Redis 分片数

大 key 处置

DEBUG OBJECT user:feed:all
Type: list | Length: 500,000 elements
Serialized length: 256 MB

发现 500 万元素的超大 List key，需要安全删除。
选择正确的删除方式：

DEL user:feed:all — 同步删除，阻塞 Redis 直到完成
UNLINK user:feed:all — 异步删除，后台线程处理，不阻塞
用 LTRIM 分批截断，每次删除 1000 条，逐步清空
等 key 自然过期，设置 TTL=1s 让它超时删除

提示：256MB 的 List，DEL 需要释放每个元素的内存，可能阻塞几十毫秒。

这是哪种故障？选择根因及处置方案：

① 故障类型：

缓存穿透 — 大量请求查询永远不存在的 key
缓存击穿 — 热点 key 过期，大量并发直接打到 DB
缓存雪崩 — 大量 key 同时过期

② 最佳处置方案：

布隆过滤器（Bloom Filter）拦截不存在的 key
singleflight 合并并发请求，只让一个请求回源 DB
为所有 key 设置随机过期时间（TTL 抖动）
