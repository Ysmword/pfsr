### 🔴 1. Go 运行时（GMP + GC）

> **为什么第一**：Go 后端中高级面试几乎必问，是拉开候选人差距的核心题，背景知识无法临时发挥。

**核心考点**

| 考点 | 关键词 |
|------|--------|
| GMP 模型 | G/M/P 定义 / Work Stealing / Hand Off / G0M0 |
| GC 演进 | 标记清除 → 三色并发 → 混合写屏障 |
| 三色标记 | 白/灰/黑 / 插入屏障 / 删除屏障 |
| 混合写屏障 | 栈对象全黑 / 堆删除变灰 / 无需 STW |
| 逃逸分析 | 栈 vs 堆 / 逃逸场景（返回指针/闭包/channel） |
| 内存泄漏 | Goroutine 泄漏 / channel 泄漏 / pprof |

**题目来源**：[面试题目.md L593](面试题目.md#L593)

**推荐游戏形式 A**：「Go GC 三色标记模拟器」（类似 I/O 多路复用模拟器）
- 亲手走一遍三色标记流程：白→灰→黑
- 对比三代 GC 的 STW 时机差异

**推荐游戏形式 B**：「GMP 调度器模拟」
- 可视化 G/M/P 的绑定关系
- 模拟系统调用阻塞时 P 的转移、Work Stealing 过程

### 🔴 2. Go 并发陷阱审判（sync + channel + defer + context）

> **为什么合并**：sync 包、channel、defer、context 四类并发陷阱在面试中往往混合出现，合并为一个游戏覆盖更完整，也避免游戏内容碎片化。

**核心考点**

| 类别 | 考点 | 关键词 |
|------|------|--------|
| sync 包 | Mutex / RWMutex | 加锁流程 / 饥饿模式 / 读写锁升级陷阱 |
| sync 包 | sync.Once | 单例实现 / 为什么不用 if 判断 |
| sync 包 | sync.Map | 适用场景 / 与加锁 map 的对比 |
| sync 包 | atomic | CAS 原理 / 适用场景 |
| sync 包 | WaitGroup | 常见误用（Add 在 goroutine 内） |
| 内存模型 | happens-before | 可见性 / 指令重排 |
| channel | 内部结构 | buf / sendq / recvq / mutex |
| channel | 读写流程 | 直接传递 / 缓冲读写 / 阻塞挂起 |
| defer | 执行顺序 | LIFO / panic-recover 范围 |
| select | 多路复用 | 随机选择 / 配合 context 超时 |

**题目来源**：[面试题目.md L787](面试题目.md#L787)（Channel 章节）

**推荐游戏形式**：「并发 Bug 代码审判」
- 给出一段 Go 代码，玩家判断：有没有 bug？数据竞争？死锁？Goroutine 泄漏？panic？
- 关卡按类型分组：sync 陷阱 → channel 陷阱 → defer 顺序 → select 边界
- 重点覆盖：RWLock 升级、WaitGroup 误用、向已关闭 channel 写入、nil channel 行为等经典坑

### 🔴 3. Go Context 与取消机制

> **为什么第一批**：生产代码中无处不在，泄漏场景面试必考。

**核心考点**

| 考点 | 关键词 |
|------|--------|
| Context 树结构 | WithCancel / WithTimeout / WithDeadline / WithValue |
| 取消传播 | 父取消 → 子取消 / 反向不传播 |
| Goroutine 泄漏 | 忘记 cancel / 阻塞在 channel 无人关闭 |
| 正确用法 | 不要存在 struct / 不要传 nil / 链路追踪 |

**推荐游戏形式**：「协程泄漏侦探」
- 给出包含 context / channel / goroutine 的代码片段
- 玩家判断：哪个 goroutine 会泄漏？为什么？怎么修？
- 关卡层层递进：超时未取消 → 多级 cancel 传播 → select + ctx 联合使用

### 🔴 5. Redis 场景题

**核心考点**

| 场景 | 解决思路 |
|------|--------|
| 热 key 问题 | 本地缓存 / 集群分片 / 探测隔离 |
| 大 key 问题 | 压缩 / 拆分 / UNLINK 异步删除 |
| 连续登录 N 天 | Bitmap 实现 |
| 实时排行榜 | ZSet 分桶 |
| 共同好友 | Set 交集（大数据量另论） |

**题目来源**：[面试题目.md L7](面试题目.md#L7)（场景题章节）

**推荐游戏形式**：「故障诊断室」（原「方案选择」→ 改为诊断式）
- 给出流量曲线 + 错误日志 + 慢查询记录，玩家先**诊断**：是热 key、大 key 还是缓存穿透？
- 诊断正确后再选解决方案，避免变成纯选择题
- 每关有「踩坑版」和「正确版」对比，揭示方案的真实代价

### 🔵 10. 分布式锁

**核心考点**

| 实现方案 | 关键机制 | 缺陷 |
|---------|---------|------|
| 数据库（乐观锁） | 版本号 + 唯一约束 | 性能低 |
| 数据库（悲观锁） | SELECT FOR UPDATE | 无自动过期 |
| Redis | SETNX + 唯一值 + 看门狗 | 主从切换丢锁 |
| RedLock | 多节点多数派写入 | 运维复杂 |
| ZooKeeper | 临时有序节点 + Watcher | 性能低（万级 QPS） |

**题目来源**：[面试题目.md L492](面试题目.md#L492)（分布式锁）

**推荐游戏形式**：「分布式锁方案对比裁判」
- 给出具体业务场景（高并发/强一致/低延迟），玩家选择最合适的实现方案
- 出现故障场景（主节点宕机/网络分区），判断各方案的行为

## 开发优先级建议

```
第一批（Go 后端核心高频）
├── Go GC 三色标记模拟器      ← 技术深度强，差距题
├── 并发 Bug 代码审判          ← sync 包，几乎必考
├── 协程泄漏侦探               ← context 取消，生产必备
└── Redis 故障诊断室           ← 热key/大key诊断式游戏

第二批（数据库深度）
├── MySQL 锁游戏              ← 死锁场景，互动性强
├── GMP 调度模拟器
└── 流量风暴生存游戏           ← 限流/熔断（⚠️ 需先整理题目）

第三批（进阶 & 补全）
├── SQL 查询路径追踪器
├── Redis 高可用故障演练
├── TCP 握手挥手模拟器
├── 分布式锁方案对比
├── MySQL 事务/日志崩溃恢复
└── MongoDB 场景选型
```
