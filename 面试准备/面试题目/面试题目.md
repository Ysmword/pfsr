#### 热key问题如何解决？

- 热key定义：在特定时间内，某个键(key)的访问量远高于其他key，从而成为访问热点。这种高频访问会给单个redis节点代理巨大压力，甚至可能导致服务过载。
- 解决方案（解决思路：分散压力、优化数据结构、保护节点）
  - 1. 本地缓存，减少 redis访问，分散redis压力
    - 优点：见效果，直接减少redis请求
    - **缺点：**
      1. 数据一致性难保证（需要处理本地缓存和redis的同步）
      2. 本地内存占用高
  - 2. redis 集群分配 -- 分散单节点压力
    - 原理：利用 redis 集群的一致性hash分片，将热key分散到不同的redis节点上，避免单个节点过载 （比如给key加前缀，分散到不同的节点）
      - 
    - 优点：横向扩展，解决单节点瓶颈
    - 缺点：需要修改应用层的分片逻辑，分片后跨节点操作复杂
  - 3. 热key探测与隔离
    - 原理：先探测热Key，再将热Key迁移到专用Redis实例​（热Key集群）或优化其存储方式。
      - **探测方法：**
        Redis内置命令​：Redis 4.0+支持hotkeys命令（需开启CONFIG SET hotkeys-enabled yes），直接返回访问频率最高的Key；
        ​监控系统​：用Prometheus+Grafana监控Redis的keyspace_hits指标，识别高频Key；
        ​自定义埋点​：在应用层记录Key的访问次数，上报到监控系统。
      - 隔离方案：将热Key迁移到独立Redis实例​（如hotkey-redis），避免影响其他业务；
    - 优缺点
      - 优点：精准定位热key，正对性优化
      - 缺点：探测工具需要额外维护，隔离增加运维成本
  - 4. 预热与缓存填充—— 避免缓存穿透​
    - 原理：提前将热Key的数据加载到Redis中（如活动开始前加载商品信息），避免活动期间大量请求穿透到数据库​（导致数据库压力骤增）
    - 优缺点
      - 优点：避免缓存穿透，提高缓存命中率；
      - **缺点：**
        1.需提前知道热Key的数据；
        2.预热失败会导致活动期间Redis压力大。

#### redis 大key问题如何解决？

- 定义：大key问题是指单个key对应的value值过大（内存占用超过10M、元素数量超百万），导致redis性能下降、内存占用激增甚至服务不可用
- 影响：
  - 1. 内存占用过高
  - 2. 网络与io瓶颈
    - **​读写延迟​：大Key的网络传输（如GET一个10MB的String）或磁盘IO（持久化时写入RDB/AOF）会阻塞其他请求，导致响应超时。**
      ​带宽占用​：大Key的传输会挤占Redis节点的网络带宽，影响其他Key的访问。
  - 3.影响持久化和恢复异常
    - **RDB快照​：生成RDB时需fork子进程，大Key的序列化会延长fork时间（秒级甚至分钟级），导致主从复制延迟。**
      ​AOF重写​：大Key的AOF日志写入会增加磁盘IO压力，重写失败可能导致数据丢失。
  - 4.删除阻塞
    - DEL大Key会触发内存回收，阻塞Redis主线程（如删除1GB的String需数秒）。
- 解决方案
  - 预警
    - 自定义脚本扫描​：使用SCAN命令遍历所有Key，结合MEMORY USAGE（Redis 4.0+）精确计算内存占用
    - **监控系统集成​：过Prometheus+Grafana监控以下指标，实时预警大Key：**
      redis_keyspace_hits_total：Key的访问次数（高频访问的大Key需关注）。
      redis_memory_used_bytes：Redis总内存（结合Key数量判断是否存在大Key）。
  - 解决方案
    - 压缩大Key——减少内存占用​
      - **适用场景：**
        大String（如存储JSON格式的长文本）；
        大Binary数据（如图片缩略图）。
    - 渐进式删除——避免阻塞（使用UNLINK命令（Redis 4.0+）异步删除大Key，或分批次删除小分片。）
      - **适用场景​：**
        需删除大Key但不想阻塞主线程；
        大Key已拆分为多个分片。
    - 拆分大Key——分散存储​（将大Key拆分为多个小Key，降低单Key的资源占用）
      - **适用场景​：**
        大Hash（如存储用户的所有属性）；
        大List（如聊天记录）；
        大Set/ZSet（如用户的关注列表）。

- 取决于key的大小和类型。对于小key的删除，删除不耗费时间，但是删除包含大量元素的 Key，会阻塞；如果想要完全不阻塞，可以使用unlink 命令，进行异步移除。

- randomkey命令从redis中随机取key，如果获取的key已经过期了，则继续查找，直至找到没有过期的数据。如果redis中存在大量过期key，如果randomkey运气不好，一直获取到过去未删除的key，则就会陷入很长时间查找。在新版本加个重试次数限制解决。

#### 如何保证redis高可用性？

- **1.  持久化机制，并且拥有三种持久化方案：**
  - RDB：快照方式持久化，将某一个时刻的内存数据，以二进制的方式写入磁盘
  - AOF：文件追加持久化，记录所有非查询操作命令，并以文本的形式追加到文件中
  - 混合持久化：RDB+AOF
- 2. 多机器部署
  - redis主从复制
    - 定义：一个Master主节点多Slave从节点的模式，将一份数据保存在多Slave个实例中，增加副本冗余量，当某些出现宕机后，Redis服务还可以使用
    - **特点：主从库之间采用读写分离方式，即**
      读操作：主库、从库都可以执行处理
      写操作：先在主库执行，再由主库将写操作结果同步给从库
    - 同步方式
      - 全量同步
        - 同步时机：主从第一次通信
        - 同步方式
          - **1. 从库启动，向主库发送psync命令**
            2. 主库接收到psync命令后，将保存RDB文件发送给从库，发送期间会使用缓存记录后序所有写操作，从库收到数据库，会清空当前数据库，然后加载从主库获取的RDB文件
            3. 主库完成RDB文件发送后，也会把将保存发送RDB文件期间写操作的replication buffer发给从库，从库再重新执行这些操作。
      - 增量同步
        - 同步时机：主从库网络断；主库有写操作
        - **同步方式：基于环形缓冲区repl_backlog_buffer缓存区实现。**
          
          在环形缓冲区，主库会记录自己写到的位置master_repl_offset，从库则会记录自己已经读到的位置slave_repl_offset, 主库并通过master_repl_offset和slave_repl_offset的差值的数据同步到从库。
          
          主从库间网络断了， 主从库会采用增量复制的方式继续同步，主库会把断连期间收到的写操作命令，写入replication buffer，同时也会把这些操作命令也写入repl_backlog_buffer这个缓冲区，然后主库并通过master_repl_offset和slave_repl_offset的差值数据同步到从库。
          
          因为repl_backlog_buffer是一个环形缓冲区，当在缓冲区写满后，主库会继续写入，此时，会出现什么情况呢？
          
          覆盖掉之前写入的操作。如果从库的读取速度比较慢，就有可能导致从库还未读取的操作被主库新写的操作覆盖了，这会导致主从库间的数据不一致。因此需要关注repl_backlog_size参数，调整合适的缓冲空间大小，避免数据覆盖，主从数据不一致。
    - 介绍下主从复制风暴
      - **redis主节点有很多从节点，在某一个时刻如果所有从节点都要从连接主节点，那么主节点同事吧内存RDB快照发给多个从节点，这样会导致redis主节点压力非常大。**
        
        解决办法，以树形结构实现复制架构

  - redis哨兵机制
    - 作用：当主库挂了，redis写操作和数据同步无法进行，为了避免这样情况，可以在主库挂了后重新在从库中选举出一个新主库，并通知到客户端。
    - 实现主从自动切换的关键机制
      - 监控：哨兵进程会周期性地给所有的主从库发送 PING 命令，检测它们是否仍然在线运行。
      - 选主：：主库挂了以后，哨兵基于一定规则评分选选举出一个从库实例新的主库 。
        - **评分依赖：**
          
          1. 从库当前在线状态
          2. 从库优先级，通过slave-priority配置项
          3. 从库复制进度，和旧主库同步程度最接近的从库得分高，通过repl_backlog_buffer缓冲区记录主库master_repl_offset和从库slave_repl_offset相差最小高分
          4. 从库ID号
      - 通知： 哨兵会将新主库的信息发送给其他从库，让它们和新主库建立连接，并进行数据复制。同时，哨兵会把新主库的信息广播通知给客户端，让它们把请求操作发到新主库上。
    - 哨兵机制一些缺陷
      - 数据丢失-主从异步复制
        - 主库的数据还没有同步到从库，结果主库发生了故障，未同步的数据就丢失了。
      - 数据丢失-脑裂
        - **何为脑裂？当一个集群中的 master 恰好网络故障，导致与 sentinal 通信不上了，sentinal会认为master下线，且sentinal选举出一个slave 作为新的 master，此时就存在两个 master了。**
          
          此时，可能存在client还没来得及切换到新的master，还继续写向旧master的数据，当master再次恢复的时候，会被作为一个slave挂到新的master 上去，自己的数据将会清空，重新从新的master 复制数据，这样就会导致数据缺失。
  - redis cluster（集群）
    - 简单介绍下： 将数据划分为 16384（2的14次方）个哈希槽（slots），如果你有多个实例节点，那么每个实例节点将管理其中一部分的槽位，槽位的信息会存储在各自所归属的节点中。
    - 某个redis挂掉后，会怎么样？
      - 每个redis部署方式时主从部署，主挂了之后，会自动将从拉起替换；之前的主好了之后，重新变成从
    - 新增一个redis
      - **1. 启动新的redis**
        2. 让新redis加入集群
        3.迁移曹位到新主节点（命令操作即可，不一定让曹位均匀）
    - redis 集群中有一个节点完全挂了，而且这个节点的从节点也挂了
      - 结果是：整个集群服务出现严重问题，因为部分数据会写失败和读失败
      - **解决办法：**
        1. 修复故障节点（高优先级）
        2.重新分配曹位（将这部分的曹位分配到负载比较低的节点上）
        3. 添加新的节点
    - 客户端如何访问？
      - **1. 客户端连接任一实例，获取到slots与实例节点的映射关系，并将该映射关系的信息缓存在本地。**
        2. 将需要访问的redis信息的key，经过CRC16计算后，再对16384 取模得到对应的 Slot 索引。
        3. 通过slot的位置进一步定位到具体所在的实例，再将请求发送到对应的实例上。
    - 新增一个key的流程？
      - **客户端连接集群时，会得到一份集群的槽位信息；客户端会将槽位信息缓存到客户端本地。当要存储key时，使用crc16算法进行hash得到一个整数值，然后用这个整数值对16384进行取模来得到具体槽位：**
        
        hash_slot = crc16(key) mod 16384
        
        在根据槽位值和redis节点的对应关系定位到key具体落在那个redis节点上。然后进行存储
    - 场景题
      - 集群网络抖动导致频繁主从切换怎么处理？
        - 在集群中添加配置 cluster-node-timeout，表示当某个节点持续timeout的时间失联时，才可以认定该节点为故障节点，才会触发主从切换
      - redis集群为什么至少需要三个节点？
        - 因为新master的选举需要大半数的集群master节点同意才能选举成功，如果只有两个master，当其中一个挂了，是达不到选举新master的条件的
      - redis集群为什么推荐奇数个？
        - 在满足master的选举需要大半数的集群master节点同意才能选举成功下，奇数不会产生平票，从而可能导致脑裂
      - redis集群支持批量操作命令吗？
        - 支持。但是有条件，必须保证所有的key在同一个solt中，否则会失败。因此如果一定要批量操作，必须在key前面加相同的{xxx}，这样参数数据分片hash计算只会是大括号里面的值，这样能确保不同的key能够落到相同的solt。
      - lua脚本能够在集群中执行吗？
        - 行，但是也要保证操作的key在hash计算后一定是同一个slot，否则就会报错。保证计算一样的方式，根支持批量操作是一样的。

#### 使用redis实现自旋锁

- 自旋锁相关
  - 定义
    - ​​ 是一种基于“忙等待”的锁机制，用于多线程（或多进程、分布式系统）中保护共享资源的互斥访问。其核心逻辑是：当一个线程尝试获取锁时，若锁已被其他线程持有，该线程不会立即阻塞或睡眠，而是通过循环（自旋）不断检查锁的状态，直到锁被释放或达到最大等待时间。
    - 本质： ​​“不放弃CPU时间片”​，通过高频次轮询锁状态来避免线程切换的开销（如上下文切换、内核态切换）。它适用于锁持有时间极短的场景，此时自旋的代价远低于线程阻塞-唤醒的代价
  - 优点
    - **低延迟响应​：线程无需进入阻塞状态，避免了上下文切换的开销（通常需几微秒到几毫秒），获取锁的延迟极低。**
      ​实现简单​：核心逻辑是循环检查锁状态，无需复杂的线程调度管理。
      ​无系统调用依赖​：纯用户态操作（或轻量级原子指令），在操作系统层面开销小。
  - 缺点
    - **​CPU资源浪费​：若锁被长时间持有，自旋线程会持续占用CPU循环检查，导致CPU空转（尤其在竞争激烈时）。**
      ​不适用长临界区​：若锁保护的代码段执行时间较长（如IO操作、复杂计算），自旋会显著降低系统吞吐量。
  - 使用场景
    - **锁持有时间极短​（如几纳秒到几微秒）：例如内核中断处理、原子变量修改等场景。**
      ​线程数少且竞争温和​：避免大量线程同时自旋导致CPU爆炸。
      ​用户态程序或轻量级服务​：如高性能中间件、游戏服务器中的短临界区保护。
      ​分布式系统中的短任务锁​：通过Redis等外部存储实现的分布式自旋锁，保护短时间内的共享资源访问。
- 实现
  - 核心原理：当一个线程在获取锁的时候，如果锁已经被其它线程获取，那么该线程将循环等待，然后不断的判断锁是否能够被成功获取，直到获取到锁才会退出循环。
  - 实现方式：
    - **1、使用 Redis 的 SET命令的 NX（Not eXists，仅当键不存在时设置）和 PX（设置过期时间，单位毫秒）选项，确保锁的获取是原子操作**
      
      2、若 SET命令返回 nil（锁已被其他客户端持有），当前客户端进入自旋循环，重试获取（可以考虑等待，但是等待时间要极短）
      
      3.获取锁后，客户端需在操作完成后释放锁。为避免误删其他客户端的锁（如锁过期后被其他客户端获取，原客户端仍尝试释放），释放时需验证锁的值是否为自己的 unique_value。

#### redis和数据库保证数据的一致性

- 不加锁
  - 特性
    - 优点：大大加快访问速度，减少数据库压力
    - 缺点：不能保证强一致性，能保证最终一致性
  - 应用场景
    - **读多写少的情况，就非常适合，详细场景：**
      1. 商品详情页/咨讯页
      2.用户行为统计或者用户信息
  - 方案
    - 读策略
      - **实现步骤：**
        如果读取的数据命中了缓存，则直接返回数据；如果读取的数据没有命中缓存，则从数据库中读取数据，然后将数据写入到缓存，并且返回给用户
        - 问题：如果修改读策略，让redis和数据库保证强一致性可以吗？
          - 答：可以，借用版本号机制，缓存中存储一个版本号，数据库存储一个版本号，在用户读取的时候，读取缓存中版本号，然后在读取数据库版本号，如果版本号一致，则直接返回，否获取数据库数据，然后更新到缓存。但是这样使用redis就没有意义，因为还是会将大量请求打到数据库
    - 写策略
      - 写策略方案都要考虑上面的读策略，对数据一致性的影响
        - 方案1：先更新数据库，在更新缓存（并发写出现问题）
          - 不足：在并发写场景下，出现不一致的概率非常大。比如：A请求先将数据库的数据更新为1，然后在更新缓存前，请求B将数据库的数据更新为2，紧接着也把缓存更新为2，然后A请求更新缓存为1。这样就导致用户读取到的是1，而不是2，预期是2（网络延迟，或者将请求改为线程）
            - (无标题)
        - 方案2：先更新缓存，然后在更新数据库（并发写出现问题）
          - **不足：**
            1. 当更新缓存成功，但是数据库更新失败时，缓存中会存储脏数据（与数据库不一致）
            2. 并发写操作导致数据错乱。A请求先将更新缓存为1，此时请求B将缓存更新为2，数据库更新2后，A请求才将数据库改为1。此时数据库数据跟缓存数据不一致（网络延迟，或者将请求改为线程）
        - 方案3：先删除缓存，再更新数据库（并发读写导致问题，数据库更新延迟）
          - 不足：还是出现数据不一致的情况。比如：假设用户年龄时20，请求A要更新用户年龄为21，会删除缓存中的内容，这时另外一个请求B要读取这个用户的年龄，他查询缓存发现缓存未命中后，会从数据库读取到年龄为20，并且写入到缓存中，然后请求A继续更改数据库，将用户的年龄更新为21，就出现不一致了
            - (无标题)
        - 方案4：先更新数据库，然后再更新缓存（并发读写导致问题，主要是redis更新延迟）
          - **不足：还是会出现，如果出现在网络延迟的情况下，还是会出现。比如：**
            假设某个用户数据在缓存中不存在，请求A读取数据时从数据库中查询到年龄为20，在未写入缓存中时，另外一个请求B更新数据。他更新数据库中的年龄为21，并且清空缓存。这时请求A把从数据库中读到的年龄为20的数据写入缓存中。
            - (无标题)
          - 优点：因为数据库更新肯定比redis更新慢，理论上数据库更新完之后，一定会删除redis里面的数据
        - 利弊判断

#### 分别说明下击穿 雪崩 穿透？或者使用过程中redis的过程中会会有哪些问题？

- 缓存击穿
  - 定义：单个热点key过期瞬间，大量并发请求直接打到数据库，导致数据库瞬时压力过大
  - 场景：热门商品库存查询、明星突发新闻的详情页面
  - 解决方案
    - 1. 设置热点数据永不过期或者过期时间很长
    - 2.提前预热，对于秒杀场景下，可以提前预热
    - 3.加锁，简单点，在每台机器上加互斥锁，当发生过期时，加锁从数据库获取后更新到缓存；后面使用分布式锁也可以
- 缓存雪崩
  - 定义：大量的key在同一个时间段集中过期，导致请求全部涌向数据库
  - 场景：单机部署故障，redis挂掉；在主从复制架构中，如果主没有开启持久化，但是给master起了一个守护进程，在哨兵没有发挥作用时，master由于没有持久化，为空数据，而slave由于要跟master保持一致，也会清空，从而导致缓存失效，进而导致缓存雪崩
  - 解决方案
    - 1. 随机设置过期时间
    - 2.过期时间设置为永不过期
    - 3. 集群部署，解决单机故障问题
    - 4. 多级缓存，使用本地缓存+redis缓存，解决redis挂掉的问题
- 缓存穿透
  - 定义：频繁查询不存在的数据，导致请求绕过缓存直达数据库
  - 场景：恶意攻击，故意查询不存在的信息
  - 解决方案
    - 缓存控制，并且设置较短的过期时间
    - 布隆过滤器，在缓存前加布隆过滤器，过滤拦截不存在的key；可能会存在误伤的情况
    - 接口校验

#### redis6 为什么引入多线程？

- 背景：以前版本的redis纯粹的单线程模型，主要限制在网络请求的读写上。redis本身的命令执行时非常快的，但是处理大量小的请求时，网络通信尤其是数据发送和接受的处理可能会造成瓶颈
  - 以jomoqie演化历史为例，第一个任务创建执行完成后，才能创建第二个任务。如果第一个任务和第二个任务都可以同时创建，那就能够大大提高总体的执行效率
- 优点以及原因
  - 提高网络I/O
    - 在之前的纯版本单线程中，如果有非常多的小请求需要处理，网络i/o（数据包的解析和写入）可能会限制整体吞吐量。通过引入多线程，redis能够并行化网络数据的读写，从而克服这一瓶颈
      - 可以参考jomoqie
  - 提高带宽利用率
    - 通过多线程处理，可以更好的利用网络带宽，尤其是在大数据量传输情况下，能够显著减少单线程造成的等待时间。
      - 可以理解为单路，变成多路
  - 降低延迟
    - 并行处理网络请求可以减少请求排队时间（总创建任务时间），从而降低请求的平均延迟，提高redis在高并发场景下的响应速度
  - 充分利用多核CPU
    - 现代服务器通常是多核CPU架构，单线程无法充分利用多核优势，引入多线程周，可以更好的利用多核资源，提高redis的整体吞吐量
- 注意点
  - 多线程只服务于网络层，而不是在命令的执行层

#### redis中key过期了一定会立即删除吗？
很多key都已经过期了，但是为什么内存占用还是很高呢？

- redis的键过期和实际删除并不是实时的，redis采用了一种惰性删除和定期删除相结合的机制处理过期键。
  - 惰性删除
    - 机制：当客户端访问一个键时，redis会检查这个键是否过期，如果键已经过期，那么他会被访问时删除掉
    - 特点：只有被访问才会被删除
  - 定期删除
    - 机制：redis会周期性的随机抽取一部分设置了过期时间的键进行检查，并删除已过期的键
      - 实际上并不是严格按照这样机制执行，他还会受一些因素影响，减少对性能的损耗
        - 1. 执行时间超过阈值，那么中断这一次定期删除循环，以免使用过多的cPU时间
        - 2.如果这一批过期的key比例超过一个比例，就会重复执行此删除流程，以更加积极清理过期的key。相应的过期的key比例低于这个比例，就会中断这次定期删除循环，以免做过多的工作，但是获得很少的内存回收。
    - 频率：定期删除由redis的后台任务执行，大约每隔25ms进行一次扫描，过期key的比例设定值为10%
    - 配置：通过配置hz参数来调整redis的内部任务执行频率，但是这个会影响整个redis进程资源的消耗（因为他会对每个设置过期时间的key设置定时器）

#### 介绍下跳表？

- 跳表是一种概率性的数据结构，通过多层索引实现高效的查找、插入和删除操作，时间复杂度为 O(log n)。

#### 最大内存淘汰机制有哪些？

- LRU类（最近最少使用，看时间，时间相隔越久，越会被删除）
  - allkeys-lru（从所有键中淘汰最久未使用的）
    - 适用场景：热点数据集中
  - volatile-lru（仅淘汰带 TTL 的键中最久未使用的）
- LFU类（最不经常使用，淘汰一段时间内，使用次数最少的数据）
  - allkeys-lfu（淘汰所有键中频率最低的）
    - 使用场景：用户行为分析，访问频率差异大，LFU 保留高频访问的用户行为数据
  - volatile-lfu（淘汰带 TTL 的键中频率最低的）
- TTL类（过期时间优先）
  - volatile-ttl（淘汰带 TTL 的键中剩余时间最短的）
    - 使用场景：临时数据为主（如验证码）
- 随机类
  - allkeys-random（随机淘汰任意键）
  - volatile-random（随机淘汰带 TTL 的键）
- 特殊类
  - noeviction（不淘汰键，拒绝写请求）

#### LRU和LFU的区别

- 实现方式区别
  - LRU：最近最少使用，看时间，时间相隔越久，越会被删除
  - LFU：最不经常使用，淘汰一段时间内，使用次数最少的数据
- 使用场景
  - 大部分请求集中在少数键的使用LRU
  - 业务中访问频率差异显著（用户行为日志）选用LFU

#### redis为什么这么快？

- 纯内存操作。redis数据读写操作都是发生在内存，访问速度都是纳秒级别。
- 高效的IO模型。单线程时间配合io多路复用技术，让单个线程可以同时处理多个网络连接上的io事件，避免了多线程模型中的上下文切换和锁竞争问题
- 优化的内部数据结构。redis提供多种数据类型（string,list,hash,set,sort set），其内部实现采用高度优化的编码方式（压缩链表、跳表等）。redis会根据数据大小和类型动态选择最合适的内部编码，以在性能和空间效率之间取的最佳平衡
- 简介高效的通信协议。redis使用自己设计的resp协议。这个协议实现简单，解析性能好，并且二进制安全。客户端和服务端之间通信的序列号/反序列号开销小，有助于提升整体的交互速度。

#### redis和mencached的区别？

- 数据类型。redis支持丰富的数据类型，不仅仅支持简单的k/v类型的数据，还同时支持list,set,zset,hash等数据结构的存储。而mencached只简单的支持k/v数据类型
- 数据持久化。redis支持数据持久化。mencached不支持
- 集群模式支持。menchache没有原生支持集群模式，需要依赖客户端实现。redis自3.0起原生支持
- 特性支持。redis支持发布订阅模式、lua脚本、事务等功能。而mencached不支持
- 过期数据删除。memcached过期数据的删除策略只支持惰性删除。而redis支持惰性删除和定期删除
- mencached多多线程，非阻塞io复用网络模型；而redis使用单线程的多路io复用模型

#### redis应用于哪些场景？

- 缓存
- 分布式锁
- 消息队列。redis自带的list数据结构可以做简单的队列使用。redis5.0中的stream类型数据结构更加适合用来做消息队列，累屎于kafka，有主题和消费组的概念，支持消息持久化以及ACK机制
- 延迟队列。基于sorted set实现。原理：用 ZSET 存储延迟消息，以“到期时间戳”作为 Score，消息内容作为 Member；消费者轮询 ZSET 中 Score ≤ 当前时间戳的消息，取出后投递到实际消费队列。

#### redis事务

- 提供一种将多个命令请求打包的功能，然后再按顺序执行打包的所有命令，并且不会被中途打断。而且需要注意redis的事务跟数据库的事务不一样，redis的事务不满足原子性和持久性。而且事务中每一条命令都会与redis服务器进行网络交互，浪费资源。
- 事务的使用
  - multi开启事务
  - exec 执行事务
  - discard抛弃事务
  - watch监控字段（对于监控的字段，在事务外被被其他客户端修改了，则事务执行失败）
- 为什么redis不支持原子性？
  - 原子性的要求为事务里面的动作要么全部执行成功，要么全部执行失败。但是redis事务中，如果有一条命令执行失败但是不是panic的错误，还是会执行其他命令。
- 为什么redis事务不支持持久性？
  - 因为redis的持久化是有时间间隔的，即还没持久化，redis就挂了，就会丢失这些数据。

#### pipline

- 作用：一批redis命令封装成一组，然后将这一组命令一次性提交到redis服务器。
- 原生批量操作和pipeline区别
  - 原生批量命令是原子操作的，pipeline是非原子操作
  - pipeline可以打包不同命令，原生命令操作不可以
  - 原生命令是服务端支持的，pipeline 需要服务端和客户端共同实现的
- pipeline和事务的比对
  - 两个事务不会同时运行，pipeline可以同时以交错方式执行
  - redis事务每一条命令都需要发送到服务端，而pipeline只需要发送一次，请求次数更少

## 分布式锁


### 定义


#### 分布式系统中协调多节点并发访问共享资源的机制，核心目标是保证在分布式环境下同一个时刻只有一个客户端执行关键操作（保证互斥性）


### 如何实现分布式锁？


#### 数据库

- 乐观锁：使用版本号+过期时间进行实现
  - **详细实现：**
    1. 获取锁时 ，先尝试插入，失败则检查过期是否过期，并原子更新锁。
    2. 释放锁时，验证持有者身份，如果身份一样，则释放锁（必须验证持有者身份）
    - **问题：为什么可以先尝试插入锁？**
      
      因为 锁字段具有唯一性约束，如果已经存则插入失败，表征当前已经有锁了
    - **问题：这个过程需要事务吗？**
      
      需要注意的是，加锁和解锁的过程是不需要放在事务里面的，对业务的操作才会考虑是否有事务
  - 缺点
    - 1. 过期时间调优：过期时间过短会导致锁频繁失效，过长会导致死锁风险
    - 2. 重试成本增加：冲突过多时，需过多重试，性能下降
    - 3. 由于数据库是要落磁盘的，性能低
- 悲观锁：利用数据库的行锁机制实现（select ... for update）
  - 注意：在mysql中for update 是当前读，在事务1使用for update加的是排他锁，如果另外一个事物依旧使用for update进行读或者修改锁定行时，会等锁。但是不影响正常读select * from tablename
  - 缺点
    - 1. 由于数据库是要落磁盘的，性能低
    - 2. 需要手动管理，因为for update没有自动过期机制，依赖事务提交/回滚释放。如果出现网络延迟，会导致无法正常提交事务，进而无法正常释放锁

#### redis

- **实现机制：**
  
  获取锁：setnx + expire + 唯一标识 (如果key已经存在，则无法插入，如果key不存在，则可以插入，并且插入过期时间，防止忘记释放锁)；
  可以使用扩展命令实现，也可以使用 lua脚本实现
  
  释放锁：1. 过期自动释放；2. 手动释放，获取key值，判断值跟唯一标识是否一致，如果不一致，不能释放，相同值则释放（这样能够避免其他线程释放锁）
  - 有个看门狗机制：主要用于自动延长锁的过期时间，防止因业务执行时间过长导致锁提前释放，从而引发并发安全问题。默认在锁过期时间的 ​1/3​ 时触发续期（如初始过期时间 30 秒，续期间隔 10 秒）
- 缺点
  - 1. 对过期时间去值比较困难，如果过期时间过短，会导致还没执行完，其他线程就能获取到锁，造成数据不一致。过长会导致如果机器挂了之后，其他机器无法获取到锁
  - 2. 使用主从复制部署的情况下，可能会导致锁失效。比如线程A加锁成功后，但是master挂了，锁的数据没有同步到slave机器，然后slave机器变成master之后，没有加锁信息，其他机器就能够获取到锁
    - 业界使用redlock去解决这种情况，就是只有同步到一定数据量的redis机器，才会认为是加锁成功
      - redlock：是 Redis 作者 Antirez 提出的分布式锁算法，专为解决单 Redis 实例故障导致的锁失效问题。它通过多数派协议（Quorum）​​ 在 Redis 集群中实现高可靠的分布式锁。
      - **实现原理：**
        1. 部署奇数个redis主节点（注意是主节点）
        2. 获取锁：多数派写入 + 时间戳校验​
        3. 锁续期：后台守护进程自动续约​
        4.释放锁
      - **缺陷：**
        1. 多机器部署，而且每个主节点之间无任何关联，增加运维成本
        2. 主机器如果后面以主从复制部署，还是会出现同步延迟的问题
        3. 主机器如果挂了，加锁之后没有持久化成功，然后被起启之后，之前加锁就不存在，其他的客户端就能够加锁成功
- 使用场景：使用高并发场景

#### zookeeper

- **实现机制：**
  ZooKeeper 实现分布式锁的核心依赖三个特性：
  ​临时节点（Ephemeral Node）​​：节点与客户端会话绑定，会话结束（客户端崩溃/断开）时自动删除，避免死锁；
  ​有序节点（Sequential Node）​​：节点创建时自动分配递增序号，保证锁的公平性；
  ​Watcher 监听机制​：客户端可监听节点的删除事件，实现“阻塞等待锁释放”。
  
  以“获取锁”和“释放锁”两个核心操作为例：
  
  1. 获取锁：创建临时有序子节点
  ​目标​：在锁路径（如 /locks/order）下创建临时有序子节点，判断自己是否是序号最小的节点（即锁的持有者）。
  ​流程​：
  客户端在 /locks/order 下创建临时有序子节点（如 lock_00000001）；
  获取 /locks/order 下的所有子节点，按序号排序；
  若自己创建的节点是序号最小的（如 lock_00000001），则获取锁成功；
  若不是最小节点（如 lock_00000002），则监听前一个节点（lock_00000001）的删除事件，等待其释放锁。
  2. 释放锁：删除节点或会话失效
  ​主动释放​：客户端执行完业务逻辑后，主动删除自己创建的临时节点（如 lock_00000001）；
  ​被动释放​：客户端崩溃或与 ZooKeeper 断开连接，会话失效，临时节点被自动删除。
  3. 锁的流动：前一个节点释放后，下一个节点获取锁
  当 lock_00000001 被删除时，ZooKeeper 会通知监听它的客户端（原 lock_00000002）。该客户端重新检查子节点列表，若此时自己是最小节点（lock_00000002），则获取锁成功。
- 缺点
  - 1. 性能瓶颈：写操作需多数节点确认
    - ZooKeeper 的写操作（如创建节点、删除节点）需要多数节点（N/2+1）确认，QPS 通常仅 ​1万~2万​（远低于 Redis 的 10万+）。
  - 2. 实现复杂度高，相比如redis来说，很多东西都需要自己开发，比如节点创建，watcher注册和实现
- 使用场景
  - **适合强一致性、低并发的场景：**
    1. 配置中心(避免多个客户端同时修改配置)

### 介绍下内存泄漏？


#### 定义：程序在申请内存后，未能及时释放不在使用的内存空间，导致这部分内存无法被再次使用，随着时间的推移，程序占用的内存不断增长，最终导致系统资源耗尽或程序崩溃


#### 造成内存泄泄漏场景（本质是内存资源无法回收）

- 1. 全局变量：全局变量在整个程序运行期间都一直存在，如果不断向全局变量中添加数据而不进行清理，将会占用越来越多的内存。
- 2. goroutine 泄漏，在 Golang 中，启动的 goroutine 如果没有正确的退出机制，将会一直存在，占用的内存也不会被释放，如果类似的 goroutine 越来越多，就会导致内存泄露。
- 3.未关闭的资源，内核资源（如 FD、Socket）未被显式释放，导致进程无法再使用这些资源。
- 4.channel 泄漏：一直向满的ch发送数据，但是没有接收方，导致发送方goruntime无法正常关闭，导致无法进行GC

#### 避免内存泄漏

- 1. 及时释放不在使用的内存
- 2. 注意goruntine的使用，确保每个goruntine都有明确的退出条件
- 3. 打开文件、数据库连接等，如果不在使用了，使用defer进行释放

#### 如何检测内存泄漏

- 1. 内存监控预警
- 2. 使用pprof工具（研究下如何使用pprof工具）

### 介绍下逃逸分析?


#### 定义：编译器在编译阶段进行的一种静态分析技术，用于判断变量的生命周期是否会超出其定义的作用域。如果超出，则变量会被分配到堆上，否则会优先分配到栈上


#### 作用

- **栈和堆是程序运行时内存的两个主要区域，各有特点：**
  ​栈​：分配/释放速度极快（仅需移动栈指针），但变量生命周期严格限制在当前函数作用域内（函数返回时栈帧销毁，变量自动回收）。
  ​堆​：变量生命周期由 GC（垃圾回收）管理，可跨函数存活，但分配/释放成本较高（需 GC 扫描），且会增加内存碎片。
  Go 编译器通过逃逸分析，尽可能将变量分配在栈上（提升性能），仅在必要时分配到堆上（保证正确性）。
  
  特别注意：逃逸分析是go内存管理非常重要的手段

#### 逃逸分析核心原则：​变量的生命周期超出其定义时的作用域


#### 发生逃逸场景

- 1. 函数返回局部变量的指针，并且函数返回后仍需被外部使用
  - 
- 2. 闭包捕获外部变量(只要闭包函数不是使用参数的方式使用外部变量，就会发生逃逸)
  - 
- 3. 发送指针到channel(其实也是局部变量被返回给函数用了，map、切片存储指针也是一样的情况)
  - 
- 4. 栈空间不足（对于 Go 语言来说，运行时(runtime) 尝试在 goroutine 需要的时候动态地分配栈空间，goroutine 的初始栈大小为 2 KB。当 goroutine 被调度时，会绑定内核线程执行，栈空间大小也不会超过操作系统的限制），特别注意如果无法确定大小的时候，也会发生逃逸，比如：generate（n int）函数
  -

### 介绍下gc


#### 关键知识点

- STW：暂停业务运行时间
- 根对象：垃圾回收器开始标记时，不需要通过其他对象来引用，默认就被认为是活跃的对象
  - 全局变量：这些变量在程序的整个生命周期中都存在，因此它们引用的对象必须一直被保留
  - 执行栈：每个goroutine都有自己的栈，栈上包含局部变量、函数参数和返回值等。这些变量可能包含指向堆内存的指针。
  - 寄存器：寄存器中可能保存着正在被CPU使用的指针。由于程序在执行过程中，寄存器中的值随时可能变化，因此GC在扫描根对象时必须捕获当前时刻寄存器中的值。

#### gc模型演化过程

- 标记清除算法（Go1.3版本之前）
  - 整体流程：暂停业务程序运行，然后标记所有可达对象，然后清除不可达对象。这个过程都不能运行用户的程序
  - 缺点
    - 需要STW，导致程序运行出现卡顿
- 三色并发标记算法（Go1.5）
  - 介绍
    - 堆空间启动写屏障，栈空间不启动，全部扫描之后，需要重新扫描一次栈 (需要 STW，只有这个时候才会有STW)，效率普通
  - 核心概念
    - **用三种颜色表示对象的标记状态：**
      ​白色​：未被 GC 访问过（初始状态，可能存活也可能死亡）。
      ​灰色​：已被 GC 访问过，但子对象未完全扫描​（需要继续处理）。
      ​黑色​：已被 GC 访问过，且所有子对象都已扫描​（确认存活）。
  - 整体流程
    - **1. 默认所有对象颜色都标记为白色**
      2. 从根节点开始遍历所有对象，把遍历到的对象从白色几何放入灰色集合
      3. 遍历灰色集合，将灰色对象引用的对象从白色放入灰色集合，之后将此灰色对象放入黑色集合
      4. 重复3步，知道灰色对象无任何对象
      5. 清除白色对象
    - 需要注意：因为标记过程跟程序运行时同时进行的，可能会出现黑色对象引用白色对象，从而导致回收正在使用的内存导致程序运行失败。因为为了防止这种情况出现，添加了屏障机制
      - 插入屏障机制
        - 当黑色对象引用白色对象的时候，将白色对象置为灰色（满足强三色不变式：不存在黑色对象引用到白色对象）
        - 注意点：只有在堆上才会使用插入屏障机制，对于栈上，在标记完成后，还要stw，再次进行三色标记，找出白色对象。
        - 为什么不考虑在栈上开启？
          - 因为栈会被频繁修改，加屏障开销太大
      - 删除屏障机制
        - 被删除的对象如果自身为灰色或者白色，那么标记为灰色（满足弱三色不变式：保护灰色对象到白色对象路径不会断）
        - 不用区分堆和栈（引入问题：回收精度低）
  - 缺点
    - 1. 依旧需要stw时间（因为栈上的标记需要STW，最终阶段必须扫描栈，补标所有漏标的白色对象）
    - 2.回收精度低，因为要满足弱三色不变式
- Go V1.8三色并发标记+混合写屏障
  - 介绍：根三色并发标记算法的整体流程基本保持一致，加入混合写屏障之后，栈不在需要STW时间。整个过程几乎不需要STW时间
  - 整体流程：跟三色并发标记法整体流程基本一致。但是优先遍历栈，然后在遍历堆
  - 混合写屏障规则
    - 1. GC开始将栈上的对象全部扫描并标记为⿊⾊(之后不再进⾏第⼆次重复扫描，⽆需STW)
    - 2.GC期间，任何栈上创建的新对象，均为黑色
    - 3. 被删除的对象标记为灰色（堆）
    - 4.被添加的对象标记为灰色（堆）
    - 5. 屏障技术不在栈上应用
  - 优点
    - 相比于三色标记法，最终阶段仅需要补考栈上黑色对象引用的白色对象

### 介绍下GMP模型


#### 定义：Go语言并发调度的核心机制，通过 Goroutine（G）、Machine（M）、Processor（P）三个核心组件的协作，能实现了高效的轻量级并发


#### GMP三组件定义与职责

- Goruntine
  - 定义：轻量级用户态线程，是并发执行的最小单位
  - 特点
    - 内存占用小：1. 初始栈仅 2KB（可动态扩缩容，最大通常 1GB），远小于操作系统线程的 MB 级栈
    - 创建/切换成本低：由 Go 运行时调度，不涉及操作系统内核的线程切换（无需陷入内核态）
    - 协作与抢占结合：支持协作式调度（只有上一个G运行完之后，才能运行下一个G），也支持抢占式调度（防止长时间运行的 G 阻塞 M，在G01.14版本后支持）
- Machine
  - 定义：操作系统调度的线程（对应 Linux 的 pthread），是真正执行代码的物理实体。
  - 特点
    - ​绑定 CPU 核心​：操作系统调度到 CPU 核心执行
    - 依赖 P 执行 G：M 必须绑定一个 P 才能执行 Goroutine（M 是「线程」，P 是「执行上下文」）
    - 阻塞时释放 P：若 M 因系统调用、通道阻塞等原因暂停，会将绑定的 P 释放给其他 M 使用，避免 CPU 空闲
- Processor
  - 定义： Go 运行时维护的逻辑处理器，负责管理 Goroutine 的上下文和调度
    - **这里的上下文是：**
      1. 本地Goruntine队列
      2.当前运行的G
      3.全局队列的引用
      4.work strealing相关状态（要偷去P的队列）
      5. 与M的关系
  - 作用
    - 连接 G 和 M​：P 是 M 执行 G 的「中介」，M 必须绑定 P 才能从 P 的队列中获取 G 执行
    - 本地 G 队列（Local RunQueue）​​：每个 P 维护一个本地队列（默认容量 256），存储待执行的 G，减少全局锁竞争
    - 全局 G 队列（Global RunQueue）​​：所有 P 共享的全局队列，当本地队列满或空时，G 会被转移到全局队列或从全局队列获取
    - ​数量可控​：P 的数量由 GOMAXPROCS决定（默认是 CPU 核心数），决定了并发的「并行度」

#### 设计策略

- 复用线程
  - stealing机制：当一个线程没有可执行的G时，会从全局队列或其他P的本地队列中偷取G来执行。
    - 从其他的队列里面偷一半
  - hand off机制：当一个线程因G进行系统调用等阻塞时，线程会释放绑定的P，把P转移给其他空闲的M执行。
- P并行
  - 有GOMAXPROCS个P，代表最多有这么多个线程并行执行
- 抢占式调度
  - Goroutine执行超过10ms就会强制让出CPU，防止其他协程饿死。
  - 特别注意：
    - 
    - **使 Goroutine 有函数调用，抢占也不是「立刻生效」：**
      调度器检测到 Goroutine 运行超过10ms​（默认阈值）时，才会发送 SIGURG信号；
      Goroutine 收到信号后，需要跳转到预设的抢占处理函数（runtime.sigusr1），这需要一定时间；
      处理函数会尝试让 Goroutine 让出 CPU，但如果循环体没有阻塞，Goroutine 可能很快又恢复运行。
- 特殊的G0和M0
  - M0：是启动程序后的编号为 0 的主线程，这个 M 对应的实例会在全局变量 runtime.m0 中，不需要在 heap 上分配，M0 负责执行初始化操作和启动第一个 G， 在之后 M0 就和其他的 M 一样了。
  - G0：G0 是每次启动一个 M 都会第一个创建的 gourtine，G0 仅用于负责调度的 G，G0 不指向任何可执行的函数，每个 M 都会有一个自己的 G0。在调度或系统调用时会使用 G0 的栈空间。

#### 协程、线程、进程三者关系？


#### 为什么不想将本地创建的协程放到其他的线程上执行？

- 1. 增加调度开销

#### 为什么需要有P？

- 1. 本地队列减少全局竞争，提升调度效率
  - 每个 P 维护一个本地运行队列（Local Run Queue）​，存储待执行的 G。M 绑定 P 后，优先从 P 的本地队列获取 G 执行，无需加全局锁。
  - 只有当本地队列为空时，P 才会从全局队列（Global Run Queue）或其他 P 的队列中“偷取”（Work Stealing）G。这种设计大幅减少了全局锁的争用，提升了并发性能。
- 2. 负载均衡的核心载体
  - **Go 的调度器需要将 G 均匀分配到多个 CPU 核心上，避免某些核心空闲、其他核心过载。P 在此过程中扮演了“负载单元”的角色：**
    默认情况下，P 的数量等于 CPU 核心数（可通过 GOMAXPROCS调整），每个 P 对应一个逻辑核心。
    当某个 P 的本地队列过满（超过 256 个 G），会将部分 G 转移到全局队列；当全局队列也满时，会进一步分散到其他 P 的队列。
    空闲的 P 会主动从全局队列或其他 P 偷取 G，确保所有 CPU 核心都能高效工作。
- 3. 管理M的生命周期，避免线程浪费
  - **M 是操作系统线程，数量通常远大于 P（Go 会缓存空闲 M，但过多线程会导致上下文切换开销）。P 的存在可以控制活跃 M 的数量：**
    当 M 绑定的 P 没有可执行的 G 时，M 可能进入休眠或缓存状态，避免占用 CPU。
    当 G 发生阻塞（如系统调用）时，P 会被释放并绑定到其他空闲 M 上，继续执行其他 G，避免线程因阻塞而闲置

### channel相关


#### 介绍下channel

- 定义：用于在不同的goroutine之间进行通信和同步的特殊数据结构
- 类型
  - 无缓存的channel
    - 无缓冲的channel没有存储空间，因此发送操作会阻塞直到另一方准备好接收数据。如果接收操作先执行，那么接收方会阻塞直到有数据发送过来
  - 有缓存的channel
    - 有缓冲的channel有固定的存储空间，发送操作只会在缓冲区满时阻塞，接收操作会在缓冲区为空时阻塞。缓冲区的大小是在创建channel时指定的
- 特点
  - 1. 类型安全：确保发送和接收的数据类型一致
  - 2. 阻塞：当发送或接收操作没有被满足时，会阻塞当前goroutine，直到满足条件
  - 3. 有缓存：可以指定缓存区大小，当缓存区已满时发送操作会被阻塞，当缓存区为空时接收操作会被阻塞
  - 4. 可以关闭：可以使用close()函数关闭Channel，关闭后的Channel不能再进行发送操作，但可以进行接收操作
  - 5. 遵循队列的先进先出特性
- channel读写流程
  - **发送数据到channel的流程：**
    1. 加锁：首先获取channel的互斥锁，保证操作的原子性。
    2. 异常检查：
        * 若channel为nil，发送操作会永久阻塞（非阻塞select中panic）。
        * 若channel已关闭，触发panic。
    3. 直接传递：
        * 若接收等待队列recvq不为空，直接从队列中取出第一个等待的接收者sudog，将数据直接拷贝到该接收者的目标内存地址，并唤醒对应的goroutine。
    4. 缓冲写入：
        * 若缓冲区未满，将数据写入缓冲区buf的sendx位置，更新sendx和qcount。
    5. 阻塞等待：
        * 若缓冲区已满，将当前goroutine封装为sudog，加入发送等待队列sendq，释放锁并挂起。
        * 当被唤醒时（由接收者取走数据或channel关闭），检查状态并继续执行
  - **从channel中读取数据流程：**
    1. 加锁：同样先获取互斥锁。
    2. 异常处理：
        * 若channel为nil，接收操作永久阻塞（非阻塞select中返回零值）。
        * 若channel已关闭且缓冲区为空，立即返回零值。
    3. 直接接收：
        * 若发送等待队列sendq不为空，分两种情况：
            * 无缓冲channel：直接从发送者处拷贝数据。
            * 有缓冲channel：取出缓冲区头部元素，将发送者的数据放入缓冲区尾部，唤醒发送者。
    4. 缓冲读取：
        * 若缓冲区有数据，读取recvx位置的元素，更新recvx和qcount。
    5. 阻塞等待：
        * 若缓冲区为空，将当前goroutine封装为sudog，加入接收等待队列recvq，释放锁并挂起。
        * 被唤醒时（数据到达或channel关闭），处理数据并返回。
- 应用场景
  - 消息传递、并发控制、优雅终止goruntine
- 注意点
  - 1. 向已经关闭的，nil的channel写入数据，是会触发panic的

#### Channel底层是使用锁控制并发的，为什么不直接使用锁？

- 虽然在Go语言中，Channel底层实现是使用锁控制并发的，但是Channel和锁的使用场景是不同的，具有不同的优势和适用性。首先，Channel比锁更加高级和抽象。Channel可以实现多个goroutine之间的同步和数据传递，不需要程序员显式地使用锁来进行线程间的协调。Channel可以避免常见的同步问题，比如死锁、饥饿等问题。其次，Channel在语言层面提供了一种更高效的并发模型。在使用锁进行并发控制时，需要程序员自己手动管理锁的获取和释放，这增加了代码复杂度和错误的风险。而使用Channel时，可以通过goroutine的调度和Channel的阻塞机制来实现更加高效和简单的并发控制。此外，Channel还可以避免一些由锁导致的性能问题，如锁竞争、锁粒度过大或过小等问题。Channel提供了一种更加精细的控制机制，能够更好地平衡不同goroutine之间的并发性能。总的来说，虽然Channel底层是使用锁控制并发的，但是Channel在语言层面提供了更加高级、抽象和高效的并发模型，可以使程序员更加方便和安全地进行并发编程。

#### Channel为什么需要两个队列实现？

- 使用两个队列实现Channel的主要原因是为了实现异步通信。发送进程可以在发送数据之后立即继续执行其他任务，而不需要等待接收进程确认收到数据。同样，接收进程也可以在等待数据到达的同时执行其他任务。这种异步通信的实现方式可以提高系统的吞吐量和响应速度。

#### 当网卡把数据写入到内存后，网卡向cpu发出一个中断信号，操作系统便能得知有新数据到来，再通过网卡中断程序去处理数据。

- 中断程序：先将网络数据写入到对应的socket的接收缓冲区里面，然后在唤醒进程

#### socket的唯一标识是：协议，本地IP，本地监听端口，远程IP，远程端口。为什么说一个进程可以监听很多个socket。因为每一个客户端连接过来都会创建一个socket。


#### 进程在操作系统的状态：就绪、等待、运行、阻塞


### io多路复用


#### IO操作包括哪些：读写文件、网络通信、设备IO


#### 没有io多路复用之前，高并发IO处理操作：

- 阻塞io：当进程发起 I/O 请求（如读取磁盘或网络数据）时，​进程会被直接挂起，直到 I/O 操作完成（数据到达或设备就绪）
- **多进程/多线程：**
  多进程：主进程监听端口，每接受一个新连接，就通过 fork()创建子进程处理该连接。
  多线程：主线程监听端口，每接受一个新连接，创建一个线程处理。
  - **问题暴露：**
    资源消耗大​：每个进程/线程需要独立的栈空间（如 Linux 默认线程栈 8MB），连接数达到数万时，内存会被耗尽。
    ​上下文切换开销高​：CPU 需频繁切换进程/线程，时间浪费在寄存器保存、内核栈切换等操作上。
    ​可扩展性差​：受限于进程/线程数上限（如 Linux 默认 pid_max为 32768），无法处理百万级并发连接

#### 定义：允许单个线程管理多个网络连接的技术，使得服务器能够高效的处理大量的并发连接而不需要为每个链接创建一个独立线程或进程


#### 发展历程

- select
  - 原理：进程通过 select()系统调用，传入一组需要监控的文件描述符（如套接字），并指定关注的事件（可读/可写/异常）。内核返回“就绪”的描述符集合，进程遍历处理。
  - 代码实现整体逻辑：先准备一个数组（下面代码中的fds），让fds存放着所有需要监视的socket。然后调用select，如果fds中的所有socket都没有数据，select会阻塞，直到有一个socket接收到数据，select返回，唤醒进程。用户可以遍历fds，通过FD_ISSET判断具体哪个socket收到数据，然后做出处理
    - 
  - 例子解释：进程A创建了三个sock1、sock2、sock3。进程调用select后，操作系统把进程A分别加入这三个socket的等待队列中。当任何一个socket收到数据后，中断程序将唤起进程。所谓唤起进程，就是将进程从所有的等待队列中移除，加入到工作队列里面。经由这些步骤，当进程A被唤醒后，它知道至少有一个socket接收了数据。进程只需遍历一遍socket列表，就可以得到就绪的socket。
  - 缺点：
    - 每次调用select都需要将进程加入到所有监视socket的等待队列，每次唤醒都需要从每个队列中移除。这里涉及了两次遍历，而且每次都要将整个fds列表传递给内核，有一定的开销。正是因为遍历操作开销大，出于效率的考量，才会规定select的最大监视数量，默认只能监视1024个socket
    - 进程被唤醒后，程序并不知道哪些socket收到数据，还需要遍历一次。
- poll
  - 原理：用 pollfd结构数组替代 select()的位掩码，理论上支持无限数量的文件描述符（仅受内存限制）。
  - 改进：解决了 select()的描述符数量限制，但仍需遍历所有监控的描述符来检查状态（时间复杂度 O(n)），在大规模连接时（如 10 万连接）效率低下。
- epoll
  - **原理：使用三个系统调用来实现。**
    1. epoll_create:创建epoll对象。进程调用epoll_create方法时，内核会创建一个eventpoll对象，对象包含用红黑树存储所有被监控的socket以及事件，以及存放以就绪的socket
    
    2. epoll_ctl向内核添加/删除/修改监控的socket
    
    3. epoll_wait等待内核就绪队列中的事件，返回就绪的socket
  - **流程：**
    1. 创建epoll对象，eventpoll对象
    2. 维护监视列表
    3. 等待socket是否可以，比如当socket接收数据后，中断程序就会给eventpoll的就绪列表中添加socket引用。（这里可以理解为事件通知后才能运行，否则会一直卡着）
    4. 当程序执行到epoll_wait时，如果就绪列表已经引用了socket，那么epoll_wait直接返回，如果为空，则阻塞进程。
  - 优点：
    - 1. 支持海量的fd，远超select的1024限制
    - 2.高效查询就绪事件。通过内核事件表，epoll_wait直接返回就绪的 fd，时间复杂度 O(1)（无需遍历所有监控的 fd）。
    - ​3. 灵活的触发模式。LT 模式适合简单场景，ET 模式适合追求极致性能的高并发场景。
      - 触发模式
        - 1. 水平触发（Level-Triggered，LT）​​
          - **行为​：只要 fd 处于“就绪状态”（如 Socket 接收缓冲区有数据），内核会反复通知应用程序（每次 epoll_wait都会返回该 fd）。**
            ​特点​：编程简单（无需一次性处理完所有数据），但可能产生冗余通知。
            ​示例​：Socket 缓冲区有 100 字节数据，LT 模式下每次 epoll_wait都会通知可读，直到数据被读完。
        - 2. 边缘触发（Edge-Triggered，ET）​​
          - **​行为​：仅当 fd 的状态发生变化时（如从“不可读”变为“可读”），内核才通知应用程序一次。**
            ​特点​：性能更高（减少冗余通知），但要求应用程序一次性处理完所有数据​（否则遗漏的数据不会再次通知）。
            ​示例​：Socket 缓冲区有 100 字节数据，ET 模式下仅第一次 epoll_wait通知可读，若未读完剩余数据，后续不再通知。

### 索引相关


#### 定义：索引是一种用于快速查询和检索数据的数据结构，其本质可以看成是一种排序好的数据结构


#### 优缺点

- 优点
  - 1. 查询速度起飞。通过索引，数据库可以大幅减少需要扫描的数据量，直接定位到符合条件的记录，从而显著加快数据的检索速度，减少磁盘I/O次数
  - 2.保证数据的唯一性。通过创建唯一索引，可以确保表中某一列(多列组合)的值是独一无二的
  - 3.加速排序和分组。如果查询中的order by和group by子句涉及的列建有索引，数据库往往可以直接利用索引已经排好序的特性，避免额外的排序操作，从而提升性能。
- 缺点
  - 创建和维护耗时。创建索引本身需要时间，特别是对大表操作时。更重要的是，当对表中的数据进行增、删、改(DML操作)时，不仅要操作数据本身，相关的索引也必须要更新和维护。
  - 占用存储空间。索引本质上也是一种数据结构，需要以物理文件的形式存储。因此会额外占用一定的磁盘空间

#### 索引底层数据结构

- hash表：通过哈希算法，快速找到key对应的index，找到index也就找到对应的value
  - 缺点：不支持顺序和范围查询
- **二叉查找树。利用二叉树的特性：**
  1. 左子树所有节点的值均小于根节点的值
  2.右子树所有节点的值均大于根节点的值
  3.左右子树也分别时二叉查找树
  - 缺点
    - 当二叉查找树不平衡时，最坏的情况下有序插入节点，树退化成线性表，导致查询效率下降到O(N)
- 平衡二叉树。查找、插入、删除在平均和最好的情况下时间复杂度都是O(logn)
  - 缺点
    - 平衡二叉树需要频繁的进行旋转操作来保持平衡，因此会有较大的计算开销进而影响数据库写操作的性能
    - 每个树节点仅存储一个数据，而每次进行磁盘IO时只能读取一个节点的数据，如果需要查询的数据分布在多个节点上，那么需要多次IO磁盘操作
- 红黑树。通过在插入和删除节点时进行颜色变化和旋转操作，使得树始终保持平衡状态
  - 缺点
    - 相比于平衡二叉树，查询效率稍微下降，因为平衡性相差较弱，可能会导致树的高度较高，可能会导致一些数据需要进行多次磁盘IO操作才能查询到
- B树&B+树，多路平衡查找树
  - B树和B+树两者差异？
    - B树的所有节点即存放键值也存放数据，而B+树只有叶子节点存放key和data，其他内节点之存放key
    - B树的叶子结点都是独立的;B+树的叶子结点有一引用链指向与他相邻的叶子结点。
    - B树中进行范围查找时，首先找到要查找的下限，然后对B树进行中序遍历，找到查找的上限；而B+树范围查询，只需要对链表进行遍历即可。

#### 索引类型

- 按底层存储划分
  - 聚簇索引：索引结构和数据存放一起的索引，InnoDB的主键索引就属于聚簇索引
    - 优点：相比于非聚簇索引，减少一次回表，查询速度比较快
    - 缺点：更新代价大，因为不仅要更新数据，还要更新索引结构中的数据
  - 非聚簇索引：索引结构和数据分开的索引，二级索引就属于非聚簇索引。mysiam引擎不管是主键索引还是非主键索引，使用的都是非聚簇索引
    - 优点：更新代价相比于聚簇索引小，因为不需要更新索引中的数据
    - 缺点：普遍相比于聚簇索引，增加一次回表查询
- 按应用维度划分
  - 主键索引：列值唯一，不能为null，表中只有一个
  - 普通索引：仅加速查询
  - 唯一索引：列值唯一，可以为null
  - 覆盖索引：一个索引包含或者说覆盖所有需要查询的字段的值
  - 联合索引：多列值组成一个索引，专门用于组合搜索，其效率大于索引合并
    - 联合索引的使用，遵循最左前缀匹配原则
      - 定义：根据索引中的字段顺序，从左到右一次匹配查询条件中的字段，如果查询条件与索引中的最左侧字段匹配，那么就会使用索引进行过滤数据
      - 原则：一直向右匹配，直到遇到范围查询(>,<)。对于>=、<=、between以及前缀匹配like的范围查询，不会停止匹配
      - 注意点
        - 顺序不一致，查询条件中的字段跟索引字段一样、而且也不会><的判断方式，数据库会优化进行使用索引查询；
        - 从左到右能够依次匹配匹配到，就可以使用或者使用部分。比如联合索引a,b,c，查询a=1 and c=1，a=1的查询会使用索引，然后在对结果c=1进行过滤。
  - 全文索引：对文本的内容进行分词，进行搜索。目前只有char、varchar、text列上可以创建全文索引，效率较低
  - 前缀索引：对文本的前几个字符创建索引，相比于普通索引建立的数据更小，因为只取几个字符

#### mysql对索引优化

- 索引下堆
  - 定义：允许存储引擎在索引遍历过程中，执行部分where子句的判断条件，直接过滤掉不满足条件的记录。比如select * from user where zipcode='3333' and month(birthdate)=3，其中联合索引是zipcode，month。在没有索引下堆之前，只能使用zipcode的索引，获取数据之后，应用层在做过滤。使用索引下堆之后，就不用应用层做过滤，直接在存储引擎层进行过滤
  - 优点：减少回表次数，减少存储引擎层和server层的数据传输量
  - 使用范围：
    - 1. 适用于innodb和myisam引擎
    - 2. 使用执行计划是range、ref、eq_ref、reg_or_null的访问查询
    - 对于innodb表，仅用于非聚簇索引。索引下堆的目标是减少全行读取次数，从而减少io操作，对于innodb的聚簇索引，完整的记录已经读取入innodb的缓存去，在这种情况下使用索引下堆不会减少io
    - 3.子查询不能使用索引下堆，因为子查询没有索引
    - 4.存储过程不能使用索引下堆，因为存储引擎无法调用存储函数

#### 问题

- 非聚簇索引一定会进行回表查询吗？
  - 不一定，如果查询的数据，刚好是索引数据，就不需要回表（覆盖索引）
- 如何选择合适的字段创建索引？
  - 1. 不为null的字段。数据为null的字段，数据库较难优化
  - 2.被频繁查询的字段
  - 3.作为查询条件的字段
  - 4.频繁需要排序的字段
  - 5.经常用于连接的字段
- 不适合创建索引的字段
  - 1.频繁被修改的字段
- 建立索引要注意什么？
  - 1. 限制每张表上的索引数量，并不是索引越多越好，虽然提高了查询效率，但是对于修改数据效率也会造成影响
  - 2. 尽可能考虑建立联合索引而不是单列索引。索引是需要占用磁盘空间的，如果表字段过多，索引过多，当表数据到达一定的体量后，修改索引是耗费的时间也是较多的
  - 3.注意避免冗余索引。能够命中索引a,b，就肯定能够命中索引a，那么索引a就是冗余索引
  - 4.字符串类型的字段使用前缀索引代替普通索引，使用前缀索引能够占用更小的空间
- 索引失效场景有哪些？
  - 1. 在where子句中对索引列直接使用函数、计算或表达式。
    - **举例：**
      假设 name 列有索引，但是用substring函数
      select * from user where substring(name,1,3)='a.b.c‘;
      
      对数值列做计算
      select * from order where amount  * 2 > 100
    - 原因：索引存储的是列的原始值，函数或表达式会改变值的形态，数据库无法直接通过索引定位的数据，这时候需要遍历所有行的计算后匹配）
  - 2. 隐式类型的转换：查询条件中的值与索引列的类型不匹配，触发数据库隐式转换
    - 举例：user_id是varchar类型，并且给user_id建立索引。然后查询：select *m from user where user_id=123 。由于为了能将数据查询出来，他会将user_id的所有列都转化为数字，然后在跟123进行判断，这个时候毕竟已经全表读取了，就不用在走索引了
  - 3.复合索引未遵循最左匹配原则，联合索引的查询条件未从最左列开始，跳过中间列
  - 4.范围查询(>,<,between)后的列失效
  - 5.否定条件(not、!=、<>、not in）
  - 6.模糊查询以通配符开头(前缀未知)
    - 原因：索引按前缀排序（如 name索引存储 '苹果手机'、'华为手机'时，前缀是 '苹果'、'华为'）。以 %开头的查询无法利用索引的前缀有序性，只能全表扫描。
  - 7.or条件分割索引列
    - 原因：OR连接的条件中，部分列有索引，部分无索引。数据库需同时满足两个条件，若其中一个条件无法使用索引，整体可能放弃索引（因 OR需合并两个结果集，索引无法高效合并）
  - 8. 小表查询或者查询数据量过大，数据库认为回表查询过大
- 走索引是什么意思？
  - 指数据库优化器选择使用索引数据结构来定位目标数据，而非直接扫描整张表
- 走全表扫描是什么意思？
  - 不走索引数据结构。直接查询磁盘中的全部数据。
- 索引支持范围查询是如何实现的？
  - 先根据条件找到起始位置，比如 a>6，就先找到a=6的索引位置的叶子结点，然后在从次叶子结点走链表查询。
- 回表定义？
  - 走了索引数据结构，但是查询的数据不在索引数据结构中，就需要其他地方获取数据。这个从其他地方获取就表示动作为回表。其他地方是指主键索引的位置
- 场景题
  - 在不带条件查询时，是否也能走索引。比如：t1表中有a,b,c的联合索引，select a from t1；select b from t1; select c from t1是否走索引？
    - 首先不带条件查询不一定就不走索引，如果查询列完全包含在覆盖索引中，就会走索引。因为覆盖索引的页中存放的数据肯定比存放所有数据页的表数据多，因此跟磁盘交互的次数就会更少，经由优化器判断，使用覆盖索引时间成本更低，因此会使用覆盖索引
    - select a from t1；select b from t1；select c from t1都会走索引
  - t1表中有a,b,c的联合索引。select a from t1 order by a；select * from t1 order by a,b,c；select b from t1；分别是否会索引？
    - 只需要考虑查询成本，如果走索引查询成本低，则走索引，否则走全表扫描
    - select a from t1 order by a
      - 走索引，因为a在联合索引中已经是排好序的，可以直接遍历联合索引的叶子结点即可
    - select * from t1 order by a,b,c
      - 不走索引。虽然使用a,b,c建立了联合索引，但是由于联合索引是非聚簇索引，叶子结点存储的是页的地址，因此还是要回表查询。由于回表查询走io，这个耗时会比内存中排序的耗时更大，因此会考虑直接全表扫描
    - select b from t1 order by a,b,c
      - 走索引，因为已经使用a,b,c建立联合索引，也就相当于a,b,c已经排好序，直接从索引中获取b即可
- 什么是innodb中的page？
  - page是innodb存储引擎管理数据库的最小磁盘单位，一个也的大小一般是16KB。一次至少读取一页数据到内存，或者刷新一页的数据到磁盘
  - page(页的数据结构)
    - **会包含：**
      文件头部
      页面头部
      行记录
      页面目录
      - 页面目录的目的是：提高查询速度。比如一页数据存储的是1～10的数据，但是我要查询11的数据，如果我没有页目录，我得取出1～10数据一个个遍历，如果有页目录，我直接判断页目录是否会包含这个数据即可。
    - (无标题)

### 锁


#### 分类

- 共享锁
  - 读锁，事务在读取记录的时候获取共享锁，允许多个事务同时获取
- 排他锁
  - 写锁/独占锁，事务在修改记录的时候获取排他锁，不允许多个事务同时获取。如果一个记录已经加了排他锁，那其他事务不能再对这条事务加任何类型的锁。

#### 粒度

- 全局锁
  - 定义：给整个数据库加锁
  - 加锁方式：flush tables with read lock、flush tables with write lock
  - 应用场景：全库备份
- 表锁
  - 定义：给整个表加锁
  - innodb的额外拥有的表级锁
    - 意向共享锁
      - 定义：事务有意向对表中的某些记录加共享锁，加共享锁之前，必须先取的该表的意向共享锁
    - 意向排他锁
      - 定义：事务有意向对表某些记录加排他锁，加排他锁之前必须先获取该表的意向排他锁
    - 特点：意向锁是数据引擎自己维护的，用户无法动手操作意向锁。并且意向锁是相互兼容的，不会互斥
    - 作用：在加表锁前，先判断记录中是否有行锁。
- 行锁
  - 定义：给行加锁
  - innodb行锁分类
    - 记录锁
      - 定义：单个行记录上加锁
    - 间隙锁
      - 定义：锁定一个范围，不包括记录本身
    - 临键锁
      - 定义：锁定一个范围，包含记录本身，主要目的是用于解决幻读问题。记录锁只能锁住已经存在的记录，为了避免插入新记录，需要依赖间隙锁。
      - 注意：innodb默认的隔离级别（可重复度），行锁默认使用的是临键锁。但是如果操作的索引是唯一索引或主键，innodb会对临键锁进行优化，将其降级为记录锁，仅锁住索引本身，而不是范围。

#### 逻辑锁

- 乐观锁
  - 定义：并不会真正的去锁某行记录，而是通过一个版本号来实现的
- 悲观锁
  - 上面说的锁，都是悲观锁

### 事务


#### 基础知识

- 定义：逻辑上的一组操作，要么都执行，要么都不执行
- 特性：ACID
  - 原子性
    - 确保动作要么全部完成，要么完全不起作用
  - 一致性
    - 数据保持一致，就是符合业务逻辑
      - 额外补充：只有事务保证持久性、原子性、隔离性之后，一致性才能得到保证。即原子性、隔离型、持久性是手段，一致性是目的
  - 隔离性
    - 事务之间互不干扰。隔离效果是由于事务隔离级别决定
    - 隔离级别
      - 读取未提交
        - 定义：最低隔离级别，允许读取尚未提交的数据，可能会导致脏读、幻读或不可重复度。
      - 读取已提交
        - 定义：允许读取并发事务已经提交的数据，可以阻止脏读，但是幻读或不可重复读取仍可能发生。（绝大多数据库默认隔离级别）
      - 可重复读
        - 定义：对同一个字段的多次读取结果都是一致的，除非数据是被本身事务自己修改的。可以阻止脏读和不可重复读。但是幻读还是可能存在。mysql的innodb默认隔离级别，使用mvcc和临键锁很大程度上解决幻读问题。
      - 可串行化
        - 定义：最高隔离级别，完全服从ACID隔离级别，所有事务逐个执行，这样事务之间就完全不可干扰，不会出现脏读、幻读、不可重复读问题
    - 实现方式
      - 可串行化是使用锁实现的，读已提交和可重复读是基于MVCC和锁实现的。
  - 持久性
    - 事务提交之后，它对数据库的数据修改是持久的，即使数据库发生故障也不应该对其有任何影响，

#### 场景问题

- 并发事务会带来什么问题？
  - 脏读
    - 读取到还没有提交的数据，如果事务再次修改或者回滚，读到的数据并非是数据库中的数据。在读取未提交隔离级别下，存在这个问题。
  - 丢失修改
    - 在一个事务读取一个数据时，另外一个事务也访问该数据，那么第一个数据做了修改，第二个事务也做了修改，这样第一个事务修改的结果就会丢失
  - 不可重复读
    - 在一个事务内，两次读取相同的数据，第一次和第二次结果不一致。
  - 幻读
    - 幻读和不可重复读类似，他发生在一个事务读取几行数据，接着另外一个并发事务插入了一些数据，在修改后的查询中，第一个事务就会发现多了一些原本不存在的记录，就好像发生了幻读一样。
  - 问题：
    - 不可重复读和幻读的区别？
      - 不可重复读是同一个事务内多次读取同一行数据，结果不一致（因其他事务修改了该行并提交）；幻读是同一事务内多次执行相同的查询，返回的结果集不一致（因其他事务插入/删除了数据的提交）
    - 在mysql中，快照读是如何避免幻读的？
      - 使用MVCC多版本并发控制解决
    - 在mysql的可重复场景下，当前读（for update）如何避免幻读？
      - 使用间隙锁，锁住不允许其他人修改，直到事务结束
- 介绍下不可重复的隔离级别下，会出现幻读？
  - 场景1：事务内，第一个查询是快照读，第二个查询是当前读（for update）

### 日志


#### 前言

- 数据库承担的角色不仅仅是数据的存储，还有一个角色是，发生意外情况时，也能保证数据的安全性。在保证数据的安全上，是通过三个核心日志去实现。三个核心日志是：binlog、redolog、undolog

#### 三个日志介绍

- binlog（二进制日志）
  - 定义：mysql服务器层维护的一种日志，记录所有对数据库内容的修改操作（不包括查询操作），并且是顺序写
  - 归属：mysql server层
  - 核心作用：
    - 数据恢复：可以通过binlog将数据库恢复到特定时间点
    - 主从复制：mysql复制的基础，主库产生binlog，从库重放这些日志
    - 数据审计：可以用于审计数据库操作，了解谁在什么时间点做了什么操作
  - 日志格式
    - statement
      - 记录内容是sql语句原文，比如执行：update t set update_time=now() where id=1语句，则记录的是这一条执行语句，以及这条语句的事务ID
        - 优势：日志量小，节省空间
        - 劣势：某些函数可能在主从产生不同的结果
        - 适用场景：sql确定性强，就是不同时间点，执行结果都一样的情况
    - row
      - 相比于statement，还会记录具体的值，比如update t set update_time=now() where id=1，update_time会记录成插入时的具体时间
        - 优势：复制精确，不受函数不确定影响
        - 劣势：日志量大，特别是大表批量操作
        - 使用场景：对数据一致性要求高的场景
    - mixed
      - 可以存放两个种格式，但是由mysql判断，执行的sql语句是否可能引起数据的不一致。如果会不一致，存储row格式日志；如果一致则使用statement。比如update_time = now()，不同时候执行，得到的结果就会不一致，而选择使用row格式记录
        - 优势：平衡了空间和一致性需求
        - 使用场景：大多数生成环境推荐选择
  - 日志写入磁盘机制
    - 写入流程：在一个事务中，每执行一条DML(增删改)语句时，都会写一条日志到 binlog cache中，待到事务提交时，在把binlog cache写到文件系统缓存page cache中，在由后台进程将page cache中的数据写入binlog。如果事务非常大，导致binlog cache空间已经被用完，这时候会将超过的部分写入到临时的磁盘文件中，待事务提交时，cache和临时文件内容合并写入binlog文件。
      - 问题？
        - 为什么使用binlog cache，而不是直接写磁盘？
          - 每写一条就要跟磁盘一次io操作，性能就太低了。
    - 写入时机策略
      - 0：由操作系统决定何时将binlog刷盘，性能最好但最不安全，如果服务器宕机了，就会可能丢失数据
      - 1：事务提交都刷盘，最安全但性能影响最大。安全是因为只有保证刷盘成功之后，才能说明事务结束
      - N：每N个事务提交后刷盘，在安全和性能间取平衡
  - 场景问题
    - binlog是数据库的全部变更记录，必须要永久保留吗？
      - Binlog 是增量变更记录，其价值依赖于“全量备份 + Binlog”的组合。若无全量备份，仅靠 Binlog 无法恢复数据（Binlog 不包含初始数据）；若已有全量备份，仅需保留备份后的 Binlog 即可。
    - binlog文件占用空间过大，怎么处理
      - 1. 适当调整下binlog的过期时间，如果过期时间过大，就会存储一些不需要的数据。
    - 主从复制中的数据不一致？
      - **1. 切换到 row格式**
        2. 主库设置binlog的刷盘策略改为1
- redolog
  - 定义：记录修改动作，在实际的物理位置上的修改动作。可以简单理解为：数据库的数据页就像是一本厚重的账本，修改数据就像在这本账本上做笔记。直接修改账本每一页费时费力。redo日志就像是一个便携的小本子，现在小本子上快速记录在第X页做了Y修改，之后再找时间更新大账本
  - 归属：innodb存储引擎层
  - 作用：保证事务的持久性和崩溃恢复能力
  - 文件内容：记录物理修改，即在Y页的 A 偏移量 B 位置，将值从 100 改为 99
  - 工作流程：
    - **1. 事务执行时，对数据页的修改先写入内存(buffer pool)**
      2. 同时将这些修改以 物理（第X页做了Y修改）形式记录到 redo日志缓存区
      3.事务提交时，将redo 日志缓存区数据写入到文件系统的page cache层
      4. 再由一定的机制写入到page cache数据写入到磁盘
      - 写入磁盘机制
        - 0：事务提交时不刷盘，由后台线程每秒将 Redo Log Buffer 刷盘 1 次​。数据先到 Page Cache，由后台线程 fsync刷盘（可能丢失 1 秒内的多个事务数据）
        - 1：每次事务提交时，Redo Log Buffer 写入 Page Cache 并立即 fsync刷盘​。数据先到 Page Cache，然后通过 fsync强制刷到磁盘（绕过 OS 延迟，确保持久化）。
        - 2：每次事务提交时，Redo Log Buffer 写入 Page Cache，但不立即 fsync​。数据停留在 Page Cache，依赖 OS 后台线程（或崩溃前）刷盘（可能丢失 1 秒内的数据）。
    - 问题
      - 除了事务提交的时候会刷盘，还有什么场景触发刷盘（将redo log 日志写入磁盘）？
        - buffer 空间不足
        - 后台线程定时刷新（会导致事务未完成而写入到磁盘）
      - 如果事务回滚了，这些已经写入到磁盘的redo log日志，会影响数据一致性吗？
        - 不会，因为回滚的时候，也会产生redo log日志
  - 场景问题
    - 为什么需要redo log，而不是直接将修改的数据直接写入磁盘就行？
      - 1. 由于mysql获取数据，不是单条获取，而是最小的获取粒度是一页（包含多条记录），一般16KB左右。相比于redo log日志，日志量相对小，刷盘耗时小
- undolog
  - 定义：记录事务对数据修改前的燕子，是的事务能够在需要时安全回滚
  - 记录内容：插入一条记录，undo日志记录如何删除它；删除一条记录，undo日志记录如何新增它；更新一条记录，undo日志记录原始值
  - undo日志生命周期管理
    - 创建：事务修改数据时创建undo记录
    - 使用：用于事务回滚和MVCC读取
    - 标记删除：事务提交后，相关的undo日志被标记为删除
    - 真正删除：后台线程回收不在需要的undo日志，比如没有MVCC会使用
  - 存储位置：存储在表空间中
- 总结：redo log用于保证事务的持久性，undo log用于保证事务的原子性；数据库备份、主备等需要依靠binlog来同步数据，保证数据一致性

#### 三个关键日志系统工作原理？

- **1. 开始事务**
  2. 事务执行过程中，生成undo日志，并写入undo log buffer
  3. 数据修改时，生成redo日志并写入 redo log buffer；而且页生成 bin log日志写入到 binlog buffer中
  4.事务提交时，redo日志写入磁盘（不一定是磁盘哈，看配置），但是标记为prepare
  5. 生成一个
  6  将事务ID也写入到binlog buffer中，然后将binlog buffer中的信息写入磁盘（不一定是磁盘哈，看配置）
  7 在redo log中记录binlog文件，并且将状态改为commit
- 问题
  - redolog为什么要分两阶段存储？
    - 若 Binlog 写入成功但 Redo Log 未刷盘，主库崩溃后从库会同步 Binlog 导致数据不一致；若 Redo Log 刷盘但 Binlog 写入失败，主库提交后从库未同步，同样不一致。两阶段提交通过“中间状态（Prepare）”确保两者要么都成功，要么都回滚。
  - redolog两个阶段之间发生奔溃会有什么影响吗？
    - **不会有任何影响。**
      仅 Redo Log 为 Prepare，Binlog 存在​：说明 Binlog 已写入，需提交事务（补全 Commit 阶段）。
      
      ​仅 Redo Log 为 Prepare，Binlog 不存在​：说明 Binlog 写入失败，需回滚事务（通过 Undo Log）。

### MVCC


#### 基本知识

- 当前读：读取的是记录最新的版本，读取时还要保证其他并发事务不能修改当前记录，会对读取的记录加锁。操作方式：select ... lock in share more(共享锁)、select ... for update(排他锁)、insert、delete、update都是一种当前读
- 快照读：简单的select 不加锁，就是快照读，读取的是记录数据的可见版本，有可能读到的是历史数据，不加锁，非阻塞读。
  - 重点：
    - **读已提交的每次select都生成一个快照读**
      可重复读是第一个select语句生成一个快照读后，后续不在生成
- mvcc：多版本控制并发。指维护一个数据的多个版本，使得读写操作没有冲突。mvcc的具体实现，还需要依赖于数据库记录中的三个隐藏字段、undu log日志、readview
  - 记录中的三个隐藏字段
    - DB_TRX_ID
      - 最后一次修改事务ID（包括插入）
    - DB_ROLL_PTR
      - 回滚指针，指向这条记录的上一个版本，用于配合undo log，指向上一个版本
    - DB_ROW_ID
      - 隐藏主键，如果表结构没有制定主键，将会生成该隐藏字段
  - undo log
    - 回滚日志，在insert、update、delete的时候产生的便于数据回滚的日志。
    - undo log 版本链：不同的事务或相同的事务对同一条记录进行修改时，会将上个版本的记录，写到undo log中，从而形成一条版本链。链头是最新的记录，链尾时最早的记录
      - 前提知识：为什么能够创建版本链？是因为多个事务修改同一行数据的时候，会加锁，进而导致修改时顺序执行的。
  - readview
    - 定义：快照读，记录系统当前活跃的事务ID（未提交），包含4个核心字段：
      - m_ids：当前活跃的事务ID集合
      - min_trx_id：最小活跃事务ID
      - max_trx_id：分配个下个事务ID，即当前最大事务ID+1
      - creator_trx_id：readview创建者的事务ID
    - 使用规则（mvcc的实现原理）
      - 用当前事务ID trx_id（要读取那一行记录的最后一次修改的事务ID），跟readview中的4个核心字段做判断，然后判断当前的事务（创建readview的事务）能够读取的版本。
      - 判断规则（有顺序的哈）
        - 1. trx_id == creator_trx_id 时，可以访问该版本
          - **场景：**
            1. 这条记录是当前事务产生的
            2. 当前事务更新完数据之后，在读取这个数据
            
            通俗讲，就是数据是当前这个事务更改的
        - 2. trx_id < min_trx_id，可以访问该版本
          - 场景：数据已经提交，肯定能读 （trx_id已经是最新的数据了）
        - 3. trx_id >= max_trx_id，不可以访问该版本
          - 场景：说明该事务是在readview生成后才开启的；通俗将，我想要看的是，我这个时间点的数据，trx_id 都大于 我记录的最大事务ID了，对于我来说是不可见的数据
        - **4. min_trx_id <= trx_id < max_trx_id，如果trx_id不在m_ids中是可以访问该版本的**
          （这里是两个必须要满足的条件）
          - **场景：数据已经提交。这个得举反例说明才行，如果满足大小判断条件，并且在m_ids中的，相当于 数据还没有提交，因此不可以访问。**

### 分库分表


#### 背景

- 随着系统数据量的增长，若采用单数据库进行数据存储，存在以下瓶颈：数据查询效率慢，输出处理慢

#### 拆分策略

- 垂直拆分
  - 垂直分库
    - 定义：以表为依据，根据业务将不同表拆分到不同库中。比如用户表放到一个数据库、订单表放在订单数据库、商品表放在商品数据库
    - 特点
      - 1. 每个库的表结构都不一样
      - 2. 每个库的数据也不一样
      - 3. 所有库的并集是全量数据
  - 垂直分表
    - 定义：以字段为依据，根据字段属性将不同字段拆分到不同表中
    - 特点：
      - 1.每个表的结构都不一样
      - 2.每个表的数据也不一样，一般通过一列（主键/外键）关联
      - 3.所有表的并集是全量数据
- 水平拆分
  - 水平分库
    - 以字段为依据，按照一定策略，将一个库的数据才分到多个库中
    - 特点
      - 1.每个库的表结构都一样
      - 2.每个库的数据都不一样
      - 3.所有库的并集是全量数据
  - 水平分表
    - 定义：以字段为依据，按照一定策略，将一个库的数据才分到多个库中
    - 特点
      - 1.每个库的表结构都一样
      - 2.每个库的数据都不一样
      - 3.所有库的并集是全量数据
- 总结：水平拆数据，垂直拆结构

#### 实现技术

- mycat：数据库分库分表中间件，不用调整代码即可实现分库分表，支持多种语言。

### 主从复制


#### 定义：将主数据库的DDL和DML操作通过二进制日志传到从库服务器中，然后从库上对这些日志重新执行，从而使得从库和主库的数据保持同步


#### 特点：支持一台主库同时向多台从库进行复制，从库同时也可以作为其他从服务器的主库，实现链状复制


#### 优点

- 1. 主库出现问题，可以快速切换到从库服务
- 2. 实现读写分离，降低主库的访问压力
- 3.可以在从库中执行备份，以避免备份期间影响主库服务

#### 主从复制原理

- 1. master 在事务提交时，会把数据变更记录在二进制日志文件中binlog中
- 2. slave 读取主库的二进制日志文件binlog，写到从库的中继日志 relay log
- 3. slave重做终极日志中的事件，将改变放映到自己的数据库
- 知识点：iothread 用于读取主库的binlog线程；sqlthread用于读取中继日志 中记录的sql，并且执行sql语句

### 基础


#### 整数类型的unsigned属性有什么用？

- 答：unsigned属性用来表示不允许负值的无符号整数。使用unsigned属性可以将正整数的上限提高一倍，因为他不需要存储负数值。比如 tinyint unsigned类型取值范围时0～255，而普通的tinyint类型的取值范围时-128～127

#### char和varchar的区别是什么？

- **答：**
  1. char是定长字符串，varchar是变长字符串。
  
  2. char在存储时会在右边填充空格以达到指定长度，检索时会去掉空格；varchar在存储时需要用1或2个额外字节记录字符串长度。检索时不需要处理；当 VARCHAR定义的最大字节数 ≤ 255 时​：用 ​1 个字节​ 记录长度前缀。反之用2个字节。
  
  3.char 适合存储长度较短或者长度差不多的字符串，比如md5算法加密后的密码，身份证号码。varchar类型适合存储长度不确定或者差异比较大的字符串，比如用户昵称，文章标题
  
  4. char(M)和varchar(M)的M都代表能够保存的字符数的最大值，无论是字母、数字还是中文，每个都只占一个字符串。比如 “我是谁”，就代表3个字符数

#### varchar(100)和varchar(10)的区别是什么？

- **1. varchar(100)和varchar(10)都是变长类型，分别表示最多能存储100个字符数和10个字符数。**
  
  2.存储的字符串数范围不同，但二者存储相同字符串，所占用的磁盘空间其实是一样的。
  
  3. varchar(100)会消耗更多的内存，因为varchar类型在内存中操作时，通常会分配固定大小的内存快来保存值，即 使用字符类型中定义的长度。比如在进行排序时，varchar(100)是按照100这个长度来进行的，也就会消耗更多的内存

#### 为什么不推荐使用TEXT？

- **1.不能有默认值**
  2.检索效率低
  3.不能直接创建索引，需要制定前缀长度
  4.可能会消耗大量的网络和io带宽

#### datetime和timestamp的区别是什么？

- **1. datatime类型没有时区信息，timestampt和时区相关**
  2.timestamp只需要用4个字节的存储空间。但datatime需要消耗8个字节的存储空间
  3. timestamp存储时间范围小（2038-01-19），datetime能够表示9999-12-31

#### NULL和‘’区别？

- **null和‘’(空字符串)是两个完全不同的值，他们代表的不同的含义，并在数据库中有不同的行为。null代表缺失或未知的数据，而''表示一个已知存在的空字符串。主要区别是：**
  1. null代表不确定的值，null不等于null。但是distinct,group by,order by 会将null值是为相同类别处理，但是不意味着null值之间相等
  
  2. 比较运算上，任何值与null进行比较的结果都是null。如果需要判断是否为null，需要用 is null 或者 is not null
  
  3.聚合函数：大多数聚合函数（sum,sgv,min,max）会忽略null值。count(*)会统计所有行，包括包含null的值，count(列名)会统计制定列中非null值的行数。空字符串会被聚合函数计算在内。例如sum会将其视为0,min 和 max会将其视为一个空字符串。

#### 布尔类型如何表示？

- mysql没有专门的布尔类型，而是用tinyint(1)类型表示布尔值。tinyint类型可以存储0或着1，分别赌赢false和true

#### mysql存储引擎架构了解吗？

- mysql存储引擎采用的是插件式架构，支持多种引擎，我们甚至可以为不同的数据库中的表设置不同的存储引擎以适应不同场景需要。存储引擎是基于表的，不是数据库

#### myisam和innodb的区别？

- 锁：myisam只有表锁，而innodb支持行级锁
- 事务：myisam不提供事务支持，innodb支持事务
- 外键：myisam不支持外键，innodb支持外键
- 安全：myisam不支持数据库异常奔溃后安全恢复，innodb支持。innodb时基于redo log实现的
- mvcc：多版本控制，myisam不支持，innodb支持
- 索引：虽然myisam和innodb都是使用B+tree作为索引结构，但是两者实现不一样。innodb引擎中，对于主键索引，索引和数据是放在一起的。myisam是索引和数据分开存放

#### mysql基础架构你有了解吗？

- mysql是由连接器、存储缓存、分析器、优化器、执行器、存储引擎构成的。
  - 连接器：主要负责用户登录数据库，进行用户的身份认证，包括校验账号密码，权限等操作。如果权限校验通过，后续管理员将权限修改，如果链接没有断开，该用户也不会受影响
  - **分析器：分析sql语句是用来干嘛的。分析器会分为两步：**
    第一步：词法分析，即提取关键字
    第二部：语法分析，判断你输入的sql的语法是否正确
  - 优化器：制定执行方案，比如多个索引的时候该如何选择索引等。
  - 执行器：校验用户是否有执行权限，如果没有权限，就会返回错误信息，如果有权限，就会调用引擎的接口，然后返回接口执行的结果信息
  - 查询缓存：缓存我们执行的select语句以及语句的结果集。缓存匹配方式是，以key-value的形式缓存在内存中，key是查询语句，value是结果集。
  - 存储引擎：主要负责数据的存储和读取。采用可以替换的插件式架构，支持Innodb、myisam，memory等多个存储引擎。
- 整体执行流程
  - 查询：权限校验 -> 查询缓存 -> 分析器 -> 优化器  -> 执行器 -> 引擎
  - 执行：权限校验 -> 分析器 -> 执行器 -> 引擎(redo log prepare状态) -> binlog -> 引擎(redo log commit)
