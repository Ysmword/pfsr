### 1. Go GC 三色标记模拟器 `go/gc_simulator/`

- **主题色** `#10b981`（绿）
- **关卡数** 5 关 · 约 10 分钟

| 关 | 标题 | 交互形式 | 核心考点 |
|----|------|---------|---------|
| 1 | 认识三色 | 拖拽 10 个堆对象到白/灰/黑三个区域 | 三色定义 + 初始全白 |
| 2 | 标记阶段走一遍 | 点击「扫描根→灰化→子节点灰化→黑化」按钮序列，节点颜色实时变化 | 标记流程可视化 |
| 3 | 写屏障触发 | 给出代码操作（`A.ref = B`），判断插入/删除屏障是否触发 | 插入屏障 vs 删除屏障 |
| 4 | 混合写屏障判断 | 给出 6 个对象操作场景，逐一判断对象最终颜色 | 栈全黑 / 堆删除变灰 / 无需 STW |
| 5 | 三代 GC 对比 | 给出三代 GC 执行时间轴，拖拽「STW」「并发标记」「写屏障触发」事件到对应时间段，三代各完成一次后展示演进对比 | 标记清除 → 三色并发 → 混合写屏障演进 |

**交互亮点**：第 2 关做有向图（节点+箭头），玩家每次点击一个节点颜色实时变化。

### 2. 并发 Bug 代码审判 `go/concurrency_court/`

- **主题色** `#ef4444`（红）
- **关卡数** 6 关 · 约 12 分钟

| 关 | 标题 | 交互形式 | 核心考点 |
|----|------|---------|---------|
| 1 | Mutex 死锁 | 展示代码，判断：数据竞争/死锁/正常？并选择原因 | Lock/Unlock 顺序 |
| 2 | RWMutex 升级陷阱 | RLock 内再 Lock → 判断是否死锁 + 选修复方案 | 读锁升写锁必死锁 |
| 3 | WaitGroup 误用 | `go func() { wg.Add(1) }` → 判断是否 panic/数据丢失 | Add 必须在 goroutine 外 |
| 4 | Channel 关闭陷阱 | 4 段代码：向已关闭写/读已关闭/nil channel 读写/关闭已关闭 → 判断行为 | channel 经典 panic 场景 |
| 5 | defer 顺序 | 两步交互：① 判断这段代码是否会 panic（单选）→ ② 拖排三个 defer 函数的执行顺序；代码控制在 3 个 defer + 1 个 recover 以内，降低脑内模拟成本 | LIFO + recover 范围 |
| 6 | select 盲点 | nil channel 在 select 中的行为 + default 竞争 | select 随机选择规则 |

**交互亮点**：每关底部「提交判决」后显示「裁判词」——一句话解释真正原因，不只是对/错。

### 3. 协程泄漏侦探 `go/goroutine_detective/`

- **主题色** `#f59e0b`（琥珀）
- **关卡数** 5 关 · 约 10 分钟

| 关 | 标题 | 交互形式 | 核心考点 |
|----|------|---------|---------|
| 1 | 忘记 cancel | 代码高亮，点击标注泄漏位置 + 选修复方案 | `defer cancel()` 必须调用 |
| 2 | channel 无人关闭 | goroutine 等待 channel，判断谁关闭 + 何时关闭 | 发送方关闭原则 |
| 3 | WithTimeout 未感知 | goroutine 内没监听 `ctx.Done()`，判断是否泄漏 + 生命周期多长 | select ctx.Done 模式 |
| 4 | 多级取消传播 | context 树可视化，点击取消节点，观察哪些子 goroutine 退出 | 取消向下传播 / 反向不传播 |
| 5 | 综合案例 | 生产级代码（HTTP Handler + goroutine + channel + context），找出所有泄漏点 | 综合判断 |

**交互亮点**：第 4 关做 context 树，点击取消后节点逐个变灰退出，直观感受传播方向。

### 4. Redis 故障诊断室 `redis/redis_diagnosis/`

- **主题色** `#f97316`（橙红）
- **关卡数** 5 关 · 约 10 分钟

| 关 | 标题 | 交互形式 | 核心考点 |
|----|------|---------|---------|
| 1 | 症状识别 | 给出 QPS 曲线 + 慢查询日志，选诊断结论：热key/大key/缓存穿透/缓存雪崩 | 4 种故障的表现差异 |
| 2 | 热 key 处理 | 确认热key后，从 4 个方案选最优（本地缓存/分片/限流/删除），说明副作用 | 本地缓存 + 集群分片 |
| 3 | 大 key 处置 | 给出 key 大小和类型，选删除方式（DEL/UNLINK/分批 HSCAN）+ 解释原因 | UNLINK 异步删除 |
| 4 | 缓存击穿诊断 | 给出 QPS 与 DB 负载曲线：某热 key 过期瞬间 DB 负载飙升 → 判断是穿透/击穿/雪崩？选 singleflight / mutex 锁方案并说明区别 | 击穿 vs 穿透 vs 雪崩；singleflight 原理 |
| 5 | Slow log 根因分析 | 给出 5 条 slowlog 记录（含 KEYS *、HGETALL 大 key、LRANGE 0 -1 等），逐条打标签（大 key 操作 / O(N) 命令 / 不当全量扫描），最后为每条选对应优化方案 | O(N) 命令识别 + SCAN 替代策略 |

## 关卡设计原则

1. **亲手操作 > 阅读选择**：能拖拽/点击/填写的，不做单纯单选题
2. **诊断在前，方案在后**：先让玩家判断是什么问题，再选解决方案
3. **每关一个核心概念**：宁可关卡多，不要一关塞三个知识点
4. **错误反馈要有教学价值**：答错后显示的解释比答对更重要
5. **关卡递进**：同一游戏内从「识别概念」到「综合判断」，难度线性上升
6. **反馈卡片必须标准化**：每关「提交」后必须显示反馈区（答对/答错不同样式），包含一句「裁判词」解释真正原因；`completeLevel()` 只在玩家点击反馈区内的「继续」按钮后才调用，不得直接跳关。模板已预留 `#levelNFeedback` 坑位，实现时直接填入。
